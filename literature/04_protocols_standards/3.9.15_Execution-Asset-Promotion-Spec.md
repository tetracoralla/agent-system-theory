# Execution Asset Promotion Specification: Maturity Levels, Promotion Rules, and Governance Interfaces

## Abstract

This specification is the normative companion to the Execution Asset Promotion Framework (EAPF) methodology paper. Where the methodology paper explains *why* execution assets need explicit governance, this specification defines *how*: seven maturity levels (M0–M6), default promotion thresholds, split/merge/demotion rules, path compliance definitions, a standard violation-code registry, minimum component requirements for skills, agents, and multi-agent systems, and the governance interfaces a harness MUST implement to enforce these rules.

The core thesis: **result correctness alone is not enough. Systems MUST evaluate whether a task used the correct execution path.** Path compliance is the first-order evaluation criterion; output quality is the second.

### Scope

This specification applies to:

- Execution-layer policy models and their decision outputs
- Skill designers and skill reviewers
- Pipeline, orchestrator, and harness developers
- Promotion Review participants
- Management agents, review agents, and architecture leads

### Normative Keywords

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

## Agent Reading Guide

### Core Claim

Execution assets require explicit maturity levels, promotion thresholds, path compliance rules, violation codes, and review gates. A system that only checks whether a task succeeded—but not whether it used the correct execution path—lacks governance over its own organizational structure.

### Key Definitions

| Term | Definition |
|------|-----------|
| **Execution Asset** | Any callable, reviewable, upgradeable execution unit from disposable script to multi-agent system. |
| **Maturity Level (M0–M6)** | A governance tier reflecting an asset's contract strength, reuse frequency, verification depth, observability, and organizational responsibility. |
| **Promotion** | Moving an asset to a higher maturity level. |
| **Split** | Decomposing an over-broad asset into narrower, well-bounded assets. |
| **Merge** | Consolidating assets that stably co-occur into a higher-level composite. |
| **Demotion** | Reducing an asset to a lower maturity level when governance overhead exceeds structural benefit. |
| **Path Compliance** | A task completed along the highest available asset path with all required gates passed. |
| **Path Violation** | A task that bypassed a higher-level asset, skipped required validation, or degraded to a lower execution path without justification. |
| **Harness** | The runtime infrastructure responsible for asset registration, permission enforcement, protocol validation, violation capture, and training-data feedback. |
| **Skill** | An M3 execution asset: a productized, reusable capability component with manifest, schema, runtime, validator, and telemetry. |
| **Agent** | An M5 execution asset: a stateful role-bearing executor that orchestrates multiple subordinate assets. |
| **Execution Protocol** | The constrained action set (`call_skill`, `clarify`, `escalate`, `reject`) that policy models MUST emit as decisions. |

### Concept Map

```
M0 (disposable script)
 └─→ M1 (LLM-call script)
 └─→ M2 (pipeline)
      └─→ M3 (skill)
           └─→ M4 (multi-skill workflow)
           └─→ M5 (agent)
                └─→ M6 (multi-agent)

Transitions: PROMOTE | SPLIT | MERGE | DEMOTE
Governance:  Harness ↔ Execution Protocol ↔ Violation Codes ↔ Promotion Review
Evaluation:  Path Compliance (1st) → Output Correctness (2nd)
```

### Do Not Misread This As

1. **"M0 is always forbidden."** M0 assets are appropriate for genuine one-time, low-risk tasks. The specification does not ban M0; it bans *residing* at M0 when promotion conditions are met.
2. **"M6 is always better."** M6 is justified only when role separation, parallelism, mutual review, or arbitration create genuine governance value. Promoting to M6 without these conditions is an organizational mismatch.
3. **"This is only about code quality."** This specification governs *organizational structure*: which execution responsibilities belong at which maturity tier, and when the system MUST restructure.
4. **"Promotion is always upward."** Split, merge, and demotion are equally important governance actions. The goal is placing each asset at its *most appropriate* level, not the highest possible level.

### Reusable Rules

| # | Rule |
|---|------|
| R1 | A system MUST prefer the highest available maturity level that satisfies the task, not the lowest feasible one. |
| R2 | Path compliance is evaluated before output correctness. |
| R3 | Every promotion review MUST produce one of: `PROMOTE`, `SPLIT`, `MERGE`, `DEMOTE`, or `DEFER_WITH_REASON`. |
| R4 | No `PV-BYPASS-*` violation MAY be recorded as "task succeeded via different implementation." |
| R5 | No `PV-CAP-*` violation MAY be treated as a single-execution defect; it is a system design defect. |
| R6 | Skills MUST NOT be promoted to agent; agents MUST NOT be promoted to multi-agent until their own level is stable. |
| R7 | The three implementation stages (stabilize → assetize → organize) MUST NOT be reordered. |

## 1. Terms and Definitions

### 1.1 Execution Asset

An **execution asset** is any callable unit the system can invoke, review, promote, or govern. The qualitative difference between levels is not "can it do the task?" but:

- Contract strength (input/output schemas, manifest completeness)
- Reuse strength (cross-project frequency)
- Verification strength (validators, review hooks)
- Observability strength (telemetry, failure classification)
- Organizational responsibility strength (role boundaries, escalation paths)

### 1.2 Path Compliance and Path Violation

**Path compliance** holds when a task completes along the system-recognized highest available asset path with all required gates (validation, review, telemetry) passed.

**Path violation** occurs when a task that could have used a higher-level asset is degraded to a lower-level path, or when required verification, review, or promotion mechanisms are bypassed.

Default path-violation candidates:

| Behavior | Why it is a candidate violation |
|----------|--------------------------------|
| Writing a disposable script when a skill already covers the task | Bypasses existing asset |
| Fabricating parameters instead of requesting clarification | Circumvents `clarify` protocol |
| Forcing a multi-step task into a single-level asset | Avoids `escalate` protocol |
| Rewriting a skill's implementation after the skill fails | Bypasses skill runtime |
| Requesting to skip validator or review | Evades quality gates |

### 1.3 Relationship to the Methodology Paper

The companion methodology paper describes an **eight-form execution asset ladder**:

```text
temporary script -> LLM-calling script -> pipeline script -> multi-script
-> skill -> multi-skill -> agent -> multi-agent
```

This specification normalizes that conceptual ladder into **seven normative maturity levels (M0-M6)**:

| Methodology paper form | Normative level in this specification |
| --- | --- |
| temporary script | M0 Disposable script |
| LLM-calling script | M1 LLM-call script |
| pipeline script | M2 Pipeline |
| multi-script | Transitional pattern between M2 and M3 |
| skill | M3 Skill |
| multi-skill | M4 Multi-skill workflow |
| agent | M5 Agent |
| multi-agent | M6 Multi-agent |

The `multi-script` form is not removed. It is treated as a transitional implementation pattern that usually indicates one of two governance actions:

- remain at M2 while interfaces and state boundaries are stabilized;
- promote to M3 when the script cluster has a stable semantic boundary, reusable input/output contract, runtime entry point, validator, and telemetry.

This normalization prevents the specification from over-institutionalizing local script ecosystems while preserving the methodology paper's conceptual distinction.

## 2. Maturity Levels M0–M6

### 2.1 Maturity Level Overview

| Level | Name | Typical Form | Applicable When | Required Governance Elements |
|-------|------|-------------|-----------------|------------------------------|
| **M0** | Disposable script | One-time script, ad-hoc command set | One-time, low-risk, no reuse expectation | Minimal I/O description, expiry awareness |
| **M1** | LLM-call script | Single-step fuzzy-processing script | Summarization, extraction, generation, conversion | Prompt constraints, output parsing, failure classification |
| **M2** | Pipeline | Fixed-sequence step chain | 3+ step stable repeating process | Step ordering, checkpoints, retry and state |
| **M3** | Skill | Productized capability component | High-frequency, reusable, semantically stable | Manifest, schema, runtime, validator, telemetry |
| **M4** | Multi-skill workflow | Stable skill composition | Multiple skills routinely co-occur | Composition contract, dependency rules, cluster-level validation |
| **M5** | Agent | Stateful role-bearing executor | Requires orchestration, judgment, escalation, resource coordination | Lifecycle, state machine, responsibility boundary, escalation interface |
| **M6** | Multi-agent | Multi-role coordinated structure | Role separation, parallelism, mutual review, arbitration | Role protocol, context isolation, handoff artifacts, arbitration mechanism |

### 2.2 Per-Level Admission and Prohibition

#### M0 — Disposable Script

**Admitted when:**

1. The task is genuinely one-time.
2. Risk is low.
3. No cross-project reuse is expected.
4. No fuzzy language understanding or open-ended generation is required.

**MUST NOT:**

1. Be used for high-frequency repeating tasks.
2. Be used when an existing skill covers the task.
3. Be used for long-lived paths requiring formal review or validation.
4. Be disguised as a formal asset.

#### M1 — LLM-Call Script

**Admitted when:**

1. The task requires single-step fuzzy processing.
2. Input/output boundaries are relatively stable.
3. The asset is not yet ready to become a pipeline or skill.

**MUST NOT:**

1. Assume multi-step orchestration responsibility.
2. Accumulate as a private script collection without promotion.
3. Bypass an existing skill.

#### M2 — Pipeline

**Admitted when:**

1. A stable sequence of 3+ steps exists.
2. The same flow is executed repeatedly.
3. State checkpoints or retries are needed.

**MUST NOT:**

1. Carry large unbounded fuzzy capabilities.
2. Replace a skill for cross-project reuse.
3. Bundle heterogeneous processes into a single pipeline.

#### M3 — Skill

**Admitted when:**

1. A stable semantic boundary exists.
2. Reuse across multiple tasks or projects is demonstrated.
3. An input schema can be clearly defined.
4. Output or success criteria are verifiable.
5. Manifest, runtime, validator, and telemetry are (or can be) provided.

**MUST NOT:**

1. Consist of documentation only, without manifest and runtime.
2. Carry overly broad responsibilities without splitting.
3. Contain stateful role responsibilities (those belong at M5).

#### M4 — Multi-Skill Workflow

**Admitted when:**

1. Multiple skills stably co-occur.
2. Inter-skill dependency order is stable.
3. Upper-layer repeated orchestration cost is high.
4. The composition relationship itself warrants governance.

**MUST NOT:**

1. Be created merely because something "looks complex."
2. Disguise an undifferentiated large skill as a multi-skill.

#### M5 — Agent

**Admitted when ALL of the following hold:**

1. Long-lived state is required.
2. Dynamic asset selection is required.
3. Exception judgment, escalation, rollback, or resource coordination is required.
4. A stable role responsibility has formed.

**MUST NOT:**

1. Be promoted merely because context is long.
2. Inflate a mechanical dispatcher into an agent.
3. Lack lifecycle and responsibility boundary definitions.

#### M6 — Multi-Agent

**Admitted when ALL of the following hold:**

1. Clear role separation exists.
2. Sub-tasks can be parallelized.
3. Mutual review, arbitration, or context isolation is needed.
4. A single agent cannot reasonably bear all organizational responsibilities.

**MUST NOT:**

1. Be the default choice for large tasks.
2. Elevate skill deficiencies or unclear agent boundaries into multi-agent problems.
3. Operate without role protocols and handoff definitions.

## 3. Promotion, Split, Merge, and Demotion Rules

### 3.1 Promotion Thresholds

#### M0 → M1/M2

An M0 asset MUST enter promotion candidacy when ANY of:

| Condition | Threshold |
|-----------|-----------|
| Same-type disposable scripts repeat | ≥ 3 occurrences within 30 days |
| LLM prompt logic embedded in script | Any occurrence |
| Script reused by a second project | Any occurrence |
| Reviewer flags logic as "should be formalized" | Any occurrence |

- If the core need is single-step fuzzy processing → promote to **M1**.
- If the core need is a stable step chain → promote to **M2**.

#### M1/M2 → M3

An M1 or M2 asset MUST enter skill candidacy when ANY of:

| Condition | Threshold |
|-----------|-----------|
| Cross-project reuse | ≥ 2 projects |
| Same-type invocations | ≥ 5 within 30 days |
| Stable input schema exists | Yes |
| Clear output artifact or success criterion | Yes |
| Upper-layer agent can stably reference by natural language | Yes |

A pipeline or LLM-call script with a stable semantic boundary that remains at script level is **promotion debt**.

#### M3 → M4 (Multi-Skill)

| Condition | Threshold |
|-----------|-----------|
| Same skill group co-occurs | ≥ 5 times |
| Inter-skill dependency order is stable | Yes |
| Repeated orchestration cost exceeds single skill call | Yes |

#### M3 → M5 (Agent)

When ANY of: long-lived state, dynamic policy judgment, resource coordination/exception escalation, or stable role responsibility is required. If the problem is inherently "role orchestration," stacking individual skills constitutes **organizational-level lag**.

#### M5 → M6

When ALL of: role separation, independent inter-role context, parallelizable sub-tasks, mutual review or arbitration, and single-agent overload are present. Lacking these, promotion to M6 is prohibited regardless of task complexity or code volume.

### 3.2 Mandatory Promotion Review Triggers

The following conditions MUST trigger a Promotion Review; deferral with "just finish this first" is not permitted:

| # | Trigger |
|---|---------|
| 1 | An existing skill is bypassed ≥ 2 consecutive times. |
| 2 | A single skill is marked "overly broad" by review ≥ 2 times. |
| 3 | The same skill group is manually re-assembled ≥ 5 times. |
| 4 | An agent is found bearing multiple weakly related role responsibilities. |
| 5 | Multiple roles objectively exist but are compressed into a single agent. |

Promotion Review output MUST be exactly one of:

| Decision | Meaning |
|----------|---------|
| `PROMOTE` | Advance to a higher maturity level. |
| `SPLIT` | Decompose into multiple narrower assets. |
| `MERGE` | Consolidate into a higher-level composite. |
| `DEMOTE` | Reduce to a lower maturity level when governance overhead exceeds structural benefit. |
| `DEFER_WITH_REASON` | Postpone with documented rationale, owner, and deadline. |

An inconclusive review output is non-compliant.

### 3.3 Split Rules

An asset MUST enter split candidacy when ANY of:

1. The input schema contains ≥ 2 weakly related parameter clusters.
2. The output produces ≥ 2 distinct artifact types.
3. Reviews flag "non-single-responsibility" ≥ 2 consecutive times.
4. Failure modes are clearly divergent with inconsistent repair paths.
5. Users cannot describe the asset's single responsibility in one stable sentence.

**Split priority order:**

1. By input contract
2. By output artifact
3. By failure mode
4. By responsibility ownership

**Prohibited pseudo-splits:**

- Splitting by file directory without addressing capability boundaries.
- Splitting solely because source files are long.
- Post-split assets still sharing a single ambiguous entry point.

Split output MUST include: new asset candidate list, proposed schemas per candidate, retirement/shrinkage plan for the original asset, and migration risk statement.

### 3.4 Merge Rules

Assets MUST enter merge candidacy when:

1. The same group co-occurs ≥ 5 times within 30 days.
2. Inter-dependency order is stable.
3. Absence of any one member significantly impairs result delivery.
4. Upper-layer orchestration logic has begun to repeat.

Merge MUST answer four questions:

1. Is the merged asset a multi-skill workflow or an agent?
2. Which responsibilities remain in the original skills?
3. Which responsibilities elevate to the composition layer?
4. Are new cluster-level validators and review hooks needed?

**Prohibited pseudo-merges:**

- Merging weakly related skills to "look unified."
- Using merge to conceal a skill that should be split.
- Claiming multi-skill status without a composition contract and cluster-level validation.

### 3.5 Demotion Rules

An agent MUST enter demotion candidacy when ANY of:

1. Actual behavior has degraded to mechanically calling a single skill.
2. No state is maintained long-term; the agent only forwards.
3. No independent role responsibility exists; it is legacy packaging.
4. Management cost exceeds structural benefit.

Demotion targets: M4 multi-skill, M3 skill, or (in extreme cases) M2 pipeline.

**Prohibited pseudo-demotions:**

- Deleting an agent without responsibility analysis.
- Losing exception handling, state management, and review entry points after demotion.
- Mistaking "this agent was poorly designed" for "the agent concept is unnecessary."

### 3.6 Multi-Agent Exit Rules

A multi-agent structure MUST enter exit or demotion evaluation when ANY of:

1. Role boundaries have been persistently unclear.
2. No real handoff artifacts exist between roles.
3. Parallelism benefit is insufficient.
4. Mutual review/arbitration mechanisms are nominal.
5. A single agent could absorb all responsibilities without governance-quality loss.

Exit evaluation output MUST be one of:

| Decision |
|----------|
| `KEEP_AS_MULTI_AGENT` |
| `REDUCE_TO_SINGLE_AGENT` |
| `REDUCE_TO_MULTI_SKILL` |
| `RESTRUCTURE_AND_RETRY` |

Multi-agent status MUST NOT be maintained solely because it is already built, looks sophisticated, or has long contexts without genuine role separation.

## 4. Minimum Requirements for Skills, Agents, and Multi-Agent Systems

### 4.1 Skill Minimum Manifest

Every asset formally classified as M3 MUST possess a machine-readable manifest with at least:

| Field | Required | Description |
|-------|----------|-------------|
| `name` | REQUIRED | Unique skill identifier. |
| `description` | REQUIRED | Trigger semantics and usage. |
| `input_schema` | REQUIRED | Input parameter contract. |
| `output_schema` | RECOMMENDED | Output artifact contract. |
| `allowed_tools` | REQUIRED | Tool scope the skill runtime may invoke. |
| `entrypoints` | REQUIRED | Runtime entry script or command. |
| `validation_rules` | REQUIRED | Result verification rules. |
| `review_required` | REQUIRED | Whether review is needed and trigger conditions. |
| `escalation_conditions` | REQUIRED | When escalation/promotion is mandatory. |
| `telemetry_fields` | REQUIRED | Logging and failure classification fields. |

An asset without a manifest MUST NOT be classified as a skill.

### 4.2 Skill Runtime Minimum

A skill runtime MUST provide:

1. A callable entry point.
2. Explicit input assembly logic.
3. Explicit output artifact or state update.
4. A failure-return mechanism.
5. Minimum observable logging.

A documentation-only asset (e.g., a specification file without independent runtime) MUST be classified as a document asset, not a skill.

### 4.3 Validator and Review Hook

Every formal skill MUST have at least two quality gates:

**Validator** — checks: input schema compliance, output format compliance, required side effects, and obvious failures.

**Review Hook** — checks: path compliance of the result, promotion/split signal detection, and violation event flagging.

- No validator → not a verifiable asset.
- No review hook → not a governable asset.

### 4.4 Skill Telemetry

Every formal skill MUST record:

| Field | Purpose |
|-------|---------|
| `skill_name` | Asset identification |
| `invocation_id` | Unique call identifier |
| `caller` | Invoking agent or process |
| `input_hash` | Input fingerprint |
| `status` | Execution outcome |
| `error_code` | Failure classification |
| `duration_ms` | Performance baseline |
| `validator_result` | Gate pass/fail |
| `review_result` | Governance pass/fail |
| `escalation_triggered` | Whether escalation fired |

Telemetry enables: skill weakness detection, training data generation, promotion review triggering, and split/merge decision support.

### 4.5 Agent Admission

An M5 agent MUST satisfy ALL of:

1. A stable role responsibility exists.
2. A lifecycle or state machine must be maintained.
3. Dynamic invocation of multiple subordinate assets is required.
4. Exception, escalation, rollback, or resource coordination judgment is required.
5. An independent escalation interface and review entry point exist.

None of the following, alone, justify agent admission:

- Long context
- Many steps
- Large code volume
- "Looks intelligent"

### 4.6 Multi-Agent Admission

An M6 multi-agent MUST satisfy ALL of:

1. ≥ 2 clear roles with distinct responsibility boundaries.
2. Formal handoff artifacts between roles.
3. Parallelism, mutual review, or arbitration value.
4. Independent context or permission boundaries per role.
5. Single-agent structure proven insufficient.

### 4.7 Prohibited Admission Scenarios

The following MUST block agent or multi-agent admission:

- Skill boundaries are not yet stable.
- The problem is skill capacity insufficiency being disguised with a higher-level structure.
- Review/validator mechanisms are absent.
- No exception ledger or promotion review mechanism exists.
- Complexity is perceived but no role differentiation exists.

**Hard rules:**

1. Skills MUST be stable before agents are introduced.
2. Agents MUST be stable before multi-agent structures are introduced.
3. Formal role protocols MUST exist before multi-agent is enabled.

## 5. Governance Interfaces

### 5.1 Execution Protocol

The execution-layer policy model's output MUST be constrained to:

| Action | Semantics |
|--------|-----------|
| `call_skill` | Invoke a single available skill with sufficient parameters. |
| `clarify` | Request missing parameters or clarify ambiguous input. |
| `escalate` | Delegate to a higher orchestration layer (multi-skill, agent, or multi-agent). |
| `reject` | Decline the task as outside system capability. |

The policy model MUST NOT output: direct code implementations, shell execution plans, direct file-write content, free-form custom actions, or multi-step array orchestration results.

**Minimum JSON schema:**

```json
{
  "decision": "call_skill | clarify | escalate | reject",
  "skill": "string | null",
  "params": {},
  "missing": [],
  "reason_code": "string"
}
```

Constraints:

| Field | Rule |
|-------|------|
| `decision` | REQUIRED. |
| `skill` | REQUIRED when `decision` is `call_skill` or `clarify`. |
| `params` | REQUIRED. MUST be an object (may be empty). |
| `missing` | Non-empty ONLY when `decision` is `clarify`. |
| `reason_code` | REQUIRED. |

**Recommended `reason_code` values:**

| Code | Meaning |
|------|---------|
| `DIRECT_MATCH` | Single skill matches with sufficient parameters. |
| `MISSING_REQUIRED_PARAMS` | Required parameters absent; clarification needed. |
| `MULTI_SKILL_OR_SEQUENCE_REQUIRED` | Task exceeds single-skill scope; escalation needed. |
| `OUT_OF_SCOPE` | Task outside system capability. |
| `BYPASS_ATTEMPT_DETECTED` | Agent detected a request to bypass existing assets. |
| `SKIP_REVIEW_REQUESTED` | Agent detected a request to skip review. |
| `SKILL_RUNTIME_INSUFFICIENT` | Available skill runtime cannot handle the task. |

### 5.2 Harness Responsibilities

The harness MUST implement four capabilities:

#### Registration and Discovery

Identify all M0–M6 assets, their levels, status, ownership, entry points, manifests, and schemas. A system that cannot uniformly discover its assets cannot enforce "highest available first."

#### Permission Control

Default-deny for policy models:

| Restriction |
|-------------|
| No direct shell write permission. |
| No direct file write permission. |
| No direct low-level tool invocation. |
| Skill runtimes receive only minimum-required permissions. |
| Review/management layers receive only role-matched permissions. |

#### Violation Capture

The harness MUST capture and record at runtime:

| Event Type |
|------------|
| Bypassing an existing skill |
| Outputting code or shell commands instead of protocol |
| Skipping validator or review |
| Executing without clarification when parameters are missing |
| Compressing a multi-step task into a single level |
| Unauthorized low-level resource access |

All captured events MUST map to standard violation codes and be written to the exception ledger.

#### Training Feedback Interface

The harness MUST expose:

- Violation event export
- Review conclusion export
- Promotion Review conclusion export
- Skill failure-mode export
- Clarification sample export

These feeds serve: policy model retraining, constraint layer rule updates, skill manifest/runtime upgrades, and asset maturity reassessment.

## 6. Violation Code Registry

### 6.1 Primary Categories

| Category | Prefix | Scope |
|----------|--------|-------|
| Path Bypass | `PV-BYPASS-*` | Should have used a higher-level asset but fell back to a lower-level implementation. |
| Contract Deficiency | `PV-CONTRACT-*` | Schema, manifest, or runtime contract is incomplete or inconsistent. |
| Review Evasion | `PV-REVIEW-*` | Attempt to skip validator, review, or quality gate. |
| Orchestration Misleveling | `PV-ORCH-*` | Should have escalated, clarified, or split but was compressed into a lower level. |
| Capability Overreach | `PV-CAP-*` | Execution layer exceeded permission boundaries. |
| Organizational Mismatch | `PV-ORG-*` | Skill/agent/multi-agent boundaries are incorrectly set. |

### 6.2 Standard Violation Codes

| Code | Meaning | Typical Trigger |
|------|---------|-----------------|
| `PV-BYPASS-EXISTING-SKILL` | Bypassed an existing skill | Wrote a disposable script for a task a skill already covers. |
| `PV-BYPASS-SKILL-RUNTIME` | Bypassed skill runtime | Read skill documentation then rewrote the implementation. |
| `PV-BYPASS-LOWER-PATH` | Deliberately degraded to a lower level | Used a one-time script where a pipeline or skill was appropriate. |
| `PV-CONTRACT-MISSING-SCHEMA` | Missing input/output schema | Skill used without a formal contract. |
| `PV-CONTRACT-RUNTIME-MISSING` | Missing runtime entry | Documentation-only asset treated as executable. |
| `PV-CONTRACT-OVERBROAD-SKILL` | Skill responsibility too broad | Single skill carrying multiple weakly related tasks. |
| `PV-REVIEW-SKIP-REQUESTED` | Requested review skip | "Skip review, just deliver." |
| `PV-REVIEW-VALIDATOR-SKIPPED` | Validator skipped | Execution completed without formal validation. |
| `PV-ORCH-SHOULD-CLARIFY` | Should have clarified but executed directly | Missing parameters were fabricated. |
| `PV-ORCH-SHOULD-ESCALATE` | Should have escalated but forced single-level | Multi-skill task compressed into single skill. |
| `PV-ORCH-SHOULD-SPLIT` | Should have split but kept mixed asset | Over-broad skill not decomposed. |
| `PV-CAP-DIRECT-SHELL` | Policy model emitted shell commands | Output contained executable command plans. |
| `PV-CAP-DIRECT-FILE-WRITE` | Policy model wrote files directly | Output attempted unauthorized code persistence. |
| `PV-CAP-DIRECT-TOOL-USE` | Bypassed harness to invoke tools | Direct tool calls without protocol layer. |
| `PV-ORG-FAKE-AGENT` | Pseudo-agent | Mechanical forwarder packaged as agent. |
| `PV-ORG-FAKE-MULTI-AGENT` | Pseudo-multi-agent | No role separation but multi-agent label applied. |
| `PV-ORG-MISSING-HANDOFF` | Missing handoff artifacts | Multiple roles with no formal handoff. |

Systems MAY extend secondary sub-codes but MUST NOT modify primary prefix semantics.

### 6.3 Violation-to-Governance Action Mapping

| Category | Default Governance Action |
|----------|--------------------------|
| `PV-BYPASS-*` | Log to exception ledger + training feedback + trigger Promotion Review. |
| `PV-CONTRACT-*` | Block formal admission; require manifest/runtime/validator completion. |
| `PV-REVIEW-*` | Treat as path failure; MUST NOT close as successful. |
| `PV-ORCH-*` | Force escalation to `clarify`/`escalate`/`split` discussion. |
| `PV-CAP-*` | Immediately block execution + permission audit. |
| `PV-ORG-*` | Enter structural reassessment; demote or restructure as needed. |

**Hard rules:**

1. `PV-BYPASS-*` MUST NOT be recorded as "task succeeded, different implementation."
2. `PV-REVIEW-*` MUST NOT be closed without supplementary review.
3. `PV-CAP-*` MUST be treated as a system design defect, not a single-execution defect.

## 7. Review Checklists

### 7.1 Execution Path Review

| Check Item | Yes/No |
|------------|--------|
| Was the highest available maturity level used? | |
| Was an existing skill or higher-level asset bypassed? | |
| Was `clarify` correctly chosen when parameters were missing? | |
| Was `escalate` correctly chosen for multi-step/multi-skill tasks? | |
| Were validator or review steps skipped? | |
| Did any capability overreach occur? | |
| Were any standard violation codes triggered? | |

Conclusion fields: `PATH_COMPLIANT`, `PATH_NON_COMPLIANT`, `VIOLATION_CODES`, `FOLLOWUP_ACTION`.

### 7.2 Skill Admission Review

Before an asset is promoted to M3, review MUST confirm:

- [ ] Single stable semantic boundary
- [ ] Manifest present
- [ ] Input schema defined
- [ ] Runtime entry point exists
- [ ] Validator present
- [ ] Review hook present
- [ ] Telemetry configured
- [ ] High-frequency reuse evidence
- [ ] No longer appropriate at script/pipeline level

### 7.3 Agent Admission Review

Before promotion to M5, review MUST confirm:

- [ ] Role responsibility exists
- [ ] State machine is required and defined
- [ ] Dynamic multi-asset orchestration is required
- [ ] Exception/escalation/rollback judgment is required
- [ ] Clear lifecycle defined
- [ ] Escalation interface exists
- [ ] Multi-skill governance scope exceeded

### 7.4 Promotion Review

Every Promotion Review MUST answer:

1. Is this a PROMOTE, SPLIT, MERGE, or DEMOTION case?
2. Is there sufficient reuse evidence or violation evidence?
3. What is the target maturity level?
4. Does the target level genuinely increase governance benefit (not just structural overhead)?
5. What new manifest/schema/runtime/review components are needed post-adjustment?
6. Do permission boundaries, violation-code mappings, or training data sampling need synchronized updates?

Promotion Review output MUST include:

| Field | Description |
|-------|-------------|
| `decision` | `PROMOTE` / `SPLIT` / `MERGE` / `DEMOTE` / `DEFER_WITH_REASON` |
| `target_level` | Target M-level. |
| `reason_codes` | Justification codes. |
| `owner` | Responsible party. |
| `deadline` | Completion deadline. |
| `required_followups` | Post-decision action items. |

"Observe and decide later" without an owner and deadline is non-compliant.

## 8. Versioning and Compatibility

### 8.1 Minimum Viable Specification

If full implementation is not immediately feasible, the minimum viable set MUST include:

1. M0–M6 maturity level table
2. Default promotion thresholds
3. Execution protocol action set
4. Minimum violation code set
5. Skill minimum component requirements
6. Harness permission restriction and violation capture

Without these six elements, a system cannot effectively prevent execution-layer degradation to disposable scripts.

### 8.2 Three-Stage Implementation Order

| Stage | Name | Actions |
|-------|------|---------|
| **Stage 1** | Stabilize | Enable execution protocol; tighten permissions; establish violation codes and exception ledger; set up minimum review checklist. |
| **Stage 2** | Assetize | Backfill skill manifest/runtime/validator/telemetry for high-frequency capabilities; launch Promotion Review; clean up over-broad skills and pseudo-agents. |
| **Stage 3** | Organize | Introduce agents for genuine role responsibilities; introduce multi-agent for genuine role differentiation; connect training feedback and governance loop. |

**The stage order MUST NOT be reversed.** In particular, multi-agent MUST NOT be introduced at scale before Stages 1 and 2 are complete.

### 8.3 Backward Compatibility

- Legacy systems MAY temporarily accept `{"skill": ..., "params": ...}` style output.
- New systems MUST migrate to the formal execution protocol.
- Existing documentation-only skills MAY operate in "quasi-skill" status temporarily but MUST NOT indefinitely defer manifest/runtime completion.

Compatibility exists for smooth migration, not for preserving low-standard paths permanently.

### 8.4 Future Machine-Readable Schema

A formal, versioned, machine-readable schema, such as JSON Schema draft-07 or later, for manifests, execution protocol payloads, and review forms is a recommended future deliverable but is not normatively required by this specification version.

## 9. Limitations and Open Questions

1. **Quantitative thresholds are heuristic.** The specific numeric thresholds (e.g., "3 occurrences in 30 days," "5 co-occurrences") are starting defaults. Systems SHOULD calibrate these based on observed data and domain characteristics.
2. **Cross-organizational comparability is untested.** Whether M0–M6 levels and violation codes transfer meaningfully across organizations with different governance cultures remains an open question.
3. **Automation depth is underspecified.** This specification defines what the harness MUST enforce but does not prescribe implementation technology, latency requirements, or error-recovery protocols for the harness itself.
4. **Training feedback loop maturity.** The specification assumes that violation events and review conclusions can be effectively fed back into policy model training. The precise curriculum design, sample weighting, and evaluation methodology for that feedback loop is outside this specification's scope.
5. **Interaction with external tool ecosystems.** When an execution asset must call external APIs or tools outside the harness's control, the boundary between "path compliant" and "unavoidable bypass" requires case-by-case judgment not fully captured by the current violation code taxonomy.
