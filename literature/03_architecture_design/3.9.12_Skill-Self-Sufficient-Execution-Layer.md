# Skill-Self-Sufficient Execution Layer for Enterprise Multi-Agent Systems

## Abstract

Enterprise multi-agent systems face a fundamental tension: small execution-layer models cannot independently perform complex code generation tasks, yet enterprises need to reduce routine dependency on cloud-based foundation model calls. This paper presents the Skill-Self-Sufficient Execution Layer architecture, a design pattern that delegates complex capabilities to self-sufficient Skills whose runtimes, schemas, validators, and optional model delegation encapsulate the actual implementation work. Under this architecture, execution-layer agents need only perform accurate Skill routing; the Skills themselves handle the cognitive heavy lifting when deeper reasoning or generation is required. We present a four-level Skill classification scheme (L0-L3), define the execution agent work loop, establish AES (Agent-Executable Specification) granularity requirements, and demonstrate how this architecture enables reliable multi-pipeline orchestration. This work extends the DCHC (Divide-and-Conquer Human-AI Collaboration) framework's L1 execution layer with self-sufficient Skill design principles, providing a practical path for enterprises deploying multi-agent automation at scale.

## Agent Reading Guide

### Core Claim

Small execution-layer models should not be asked to write code or solve complex tasks directly. They should route tasks to self-sufficient Skills whose runtimes, schemas, validators, and optional model delegation encapsulate the actual capability. The quality ceiling of the system is determined primarily by Skill quality, not execution-layer model capability.

### Key Definitions

| Term | Definition |
|------|------------|
| **Execution Layer** | The L1 layer of the DCHC framework; receives AES nodes and orchestrates their execution via Skill calls |
| **Skill** | A self-contained execution unit with standardized input/output contracts, capable of performing a defined task category |
| **Skill Routing** | The process by which an execution-layer agent matches a task to the correct Skill and extracts required parameters |
| **Self-Sufficient Skill** | A Skill that encapsulates required capabilities within the Skill boundary, including optional model delegation, without requiring the calling agent to perform the core task |
| **AES** | Agent-Executable Specification; a structured YAML document defining a single, granular task with skill bindings and validation commands |
| **AES Granularity** | The degree to which AES nodes are decomposed to the level of single Skill calls |
| **Provider Adapter** | The runtime component that manages optional foundation-model invocations from within Skills |

### Do Not Misread This As

1. **"Small models are always better."** The claim is about role separation and cost/control tradeoffs. Cloud models remain essential—they are simply invoked from within Skills rather than by execution agents directly.

2. **"Skills eliminate the need for model calls."** L3 Skills may invoke foundation models through provider adapters. The architecture moves these invocations from the execution agent to the Skill layer, reducing routing overhead and enabling offline execution-layer routing where possible.

3. **"Any AES node can be executed by any small model."** AES nodes must be decomposed to match single Skill calls. Large, multi-step AES nodes that require substantial code generation exceed small model capabilities.

4. **"Skill routing is trivial."** While smaller than code generation, Skill routing requires fine-tuned models or strong constraint layers for reliable task classification and parameter extraction.

### Reusable Rules

1. **Role Separation Principle**: "What to do" belongs to the management layer; "which Skill to use" belongs to the execution layer; "how to do it" belongs to the Skill layer.

2. **Skill Self-Sufficiency Invariant**: A Skill's output quality should not materially depend on the calling agent's model capability. If a small routing model calling a Skill produces much worse results than a frontier model calling the same Skill with the same inputs, the Skill is not self-sufficient.

3. **AES Atomicity Rule**: Each AES node should map to exactly one Skill call. Nodes requiring multi-Skill orchestration belong at a higher layer.

4. **Routing Over Generation**: Execution-layer models should be evaluated on routing accuracy (skill matching + parameter extraction), not on generation quality.

5. **Validation at Boundary**: Results should be validated at the Skill layer before returning to the execution agent, with failures bubbling up with structured error context.

## 1. Problem: Weak Execution Models and Overpowered Direct Generation

### 1.1 The Capability Mismatch

Enterprise multi-agent systems increasingly employ a three-layer architecture: a powerful management layer for planning and decomposition, a lean execution layer for orchestration, and a library of Skills for actual task execution. The economic motivation is clear: local, small-parameter models at the execution layer are usually cheaper to operate per routing decision than cloud-based frontier models, can run offline, reduce network latency for routing decisions, and preserve privacy by processing task descriptions locally.

However, this architecture encounters a fundamental capability mismatch when existing Skills are "pure prompt" artifacts—text files that instruct the calling agent to perform the actual work. Consider the following analysis of common Skills:

| Skill Category | Skill Name | Internal Mechanism | LLM Dependency |
|---------------|------------|-------------------|----------------|
| Code generation | `frontend-design` | 60 lines of design guidance text | 100% on caller |
| Code generation | `frontend-skill` | Strict design rules text | 100% on caller |
| Code generation | `develop-web-game` | Playwright tests + caller writes code | High on caller |
| API wrapper | `imagegen` | Image generation API call | Zero |
| API wrapper | `videogen` | Video generation API call | Zero |
| Deterministic | `playwright` | Runs test scripts | Zero |

The problem: many code generation Skills in prompt-only libraries produce outputs whose quality depends almost entirely on the calling model's capability. A small routing model attempting to use a prompt-only frontend Skill to generate a login page will often fail because the Skill's guidance text requires frontier-level comprehension to yield good outputs.

### 1.2 AES Granularity Mismatch

Current AES nodes often represent substantial implementation tasks. Consider a typical AES node:

| Requirement | Small Model Capable? |
|-------------|---------------------|
| Read AES YAML, extract fields | Yes |
| Apply runtime interface patterns | No; requires code understanding |
| Write substantial async code | No; requires code generation |
| Implement composition patterns | No; requires architectural reasoning |
| Run required validation commands | Yes |
| Check file existence and attributes | Yes |

A single AES node may contain some tasks a small model can perform, such as parsing, validation, and checking, alongside tasks requiring stronger models, such as design and coding. This granularity exceeds small-model scheduling and execution capability.

### 1.3 Small Model Capability Boundaries

Empirical testing on small instruction-tuned models reveals reliable capabilities in:

- Structured data extraction (reading YAML/JSON and extracting fields)
- Text classification (matching task descriptions to Skill names)
- Simple parameter mapping (extracting key parameters from task descriptions)
- Deterministic result verification (running test scripts, checking file existence)

And unreliable capabilities in:

- Zero-shot JSON format output without constraints
- Natural language to tool name mapping without fine-tuning
- Code generation (outputs SQL queries instead of JSON)

The conclusion: 1B model baseline capabilities are insufficient for direct Skill routing, but can reach acceptable performance through a combination of fine-tuning, constraint layers, and Skill self-sufficiency.

## 2. The Three-Layer Division of Labor

### 2.1 Layer Responsibilities

The enterprise multi-agent system divides responsibilities across three layers:

```
┌─────────────────────────────────────────────────────────────┐
│  Management Layer (frontier/domain models)                  │
│  Responsibilities:                                          │
│    - Architecture design                                    │
│    - Implementation planning                                │
│    - AES decomposition                                       │
│    - Workflow orchestration                                  │
│  Characteristics: High capability, higher cost, network    │
│                    required                                  │
└─────────────────────────────┬───────────────────────────────┘
                              │ Fine-grained AES nodes
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Execution Routing Layer (Local: 1B-3B + Constraint Layer) │
│  Responsibilities:                                          │
│    - Read AES                                               │
│    - Match Skill                                            │
│    - Extract parameters                                     │
│    - Call Skill                                             │
│    - Validate results                                       │
│  Characteristics: Low capability, low cost, offline-capable│
└─────────────────────────────┬───────────────────────────────┘
                              │ Skill calls
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Skill Layer (60+ Skills)                                    │
│  L0: Deterministic scripts (playwright, screenshot, yeet)    │
│  L1: API wrappers (imagegen, sora, figma, speech)           │
│  L2: Template filling (slides, algorithmic-art)             │
│  L3: LLM-delegated (frontend-design, python-codegen)       │
└─────────────────────────────────────────────────────────────┘
```

The governing principle: **Management layer decides what to do; execution layer decides which Skill to use; Skill layer handles how to do it.**

### 2.2 Why Not Direct Cloud Execution?

A reasonable objection: if Skills ultimately invoke cloud models, why not use cloud models directly at the execution layer, eliminating the local small model layer entirely?

| Dimension | Direct Cloud | Local Execution + Cloud Skill |
|-----------|--------------|-------------------------------|
| **Latency** | Every routing call over network | Routing local (<100ms), only Skill execution over network |
| **Cost** | Every routing consumes tokens | Routing is free; only Skill execution consumes tokens |
| **Offline capability** | None | Skill routing works offline (L0/L1 Skills partially offline) |
| **Privacy** | All task descriptions sent to cloud | Task descriptions processed locally only |
| **Reliability** | Depends on network and API availability | Local routing unaffected by network issues |
| **Scalability** | Subject to API rate limits | Local routing has no rate limits |

The execution layer's value lies not in replacing cloud models but in **stripping high-frequency, low-value "routing decisions" from the cloud**. Analogous to an API gateway in microservices—gateways perform no business logic but reduce backend service load through local routing.

## 3. The Skill Self-Sufficiency Model

### 3.1 Four-Level Skill Classification

Based on analysis of 60+ Skills, we establish a four-level classification scheme based on LLM dependency:

#### L0: Deterministic Scripts (Zero LLM Dependency)

Skills in this category execute entirely via deterministic code with no LLM invocations.

| Skill | Function | Input | Output |
|-------|----------|-------|--------|
| `playwright` | Browser automation | URL + operation sequence | Screenshot/log |
| `screenshot` | System screenshot | Region parameters | Image file |
| `yeet` | Git commit + PR creation | File list + message | PR URL |
| `jupyter-notebook` | Notebook creation | Template parameters | `.ipynb` file |
| `transcribe` | Audio transcription | Audio file path | Text |

**Transformation needs**: None. These Skills are already self-sufficient.

#### L1: API Wrappers (External Service Calls, Zero LLM Dependency)

Skills in this category encapsulate third-party API calls.

| Skill | Function | External API |
|-------|----------|--------------|
| `imagegen` | Image generation | Image generation API |
| `videogen` | Video generation | Video generation API |
| `speech` | Text-to-speech | Speech generation API |
| `figma` | Figma operations | Figma MCP Server |
| `pdf` | PDF generation/parsing | Poppler/ReportLab |
| `xlsx` | Spreadsheet operations | openpyxl |
| `docx` | Word document operations | python-docx |
| `pptx` | Slide operations | PptxGenJS |
| `cloudflare-deploy` | Deploy to Cloudflare | CF Workers API |
| `vercel-deploy` | Deploy to Vercel | Vercel API |

**Transformation needs**: None, beyond adding standardized input/output schemas.

#### L2: Template Filling (Template + Minor LLM)

Skills in this category use predefined templates with minor LLM capability for parameter filling.

| Skill | Function | Template Type |
|-------|----------|---------------|
| `slides` | Slide generation | Layout template + content filling |
| `algorithmic-art` | Algorithmic art | p5.js template + parameter adjustment |
| `web-artifacts-builder` | Web application generation | React scaffold + component filling |

**Transformation needs**: Moderate. The model-assisted filling portion should be encapsulated as a provider-adapter call within the Skill.

#### L3: Provider-Delegated Skills

This category is the **primary transformation target**. Current Skills in this category are "pure prompt" artifacts; after transformation, they invoke provider adapters when model delegation is needed.

| Skill | Function | Current State | After Transformation |
|-------|----------|---------------|---------------------|
| Frontend generation Skill | Frontend code generation | Pure prompt guiding caller | Provider-adapter call |
| Landing-page Skill | Landing page generation | Pure prompt guiding caller | Provider-adapter call |
| Web-game Skill | Web game development | Prompt + browser validation | Provider adapter + browser validation |
| Backend framework Skill | Backend development | Pure prompt | Provider-adapter call |
| Desktop app Skill | Desktop application development | Pure prompt | Provider-adapter call |
| Security review Skill | Security review | Pure prompt | Provider-adapter call |

**Transformation needs**: High. These Skills require architectural refactoring.

### 3.2 Core Transformation: L3 Skills from "Pure Prompt" to "Provider Delegation"

#### Before (Problem Architecture)

```
Small model reads Skill → Skill is pure text guidance → Small model writes code per guidance → Unreliable output
```

#### After (Target Architecture)

```
Small model matches Skill → Passes parameters → Skill invokes provider adapter when needed → Validated output
```

#### The Transformation Pattern

**Step 1: Add a standardized executor module to each L3 Skill**

Each self-sufficient Skill implements an executor class inheriting from a base class:

```
Skill directory structure:
  skill-name/
  ├── SKILL.md           # Metadata + human-readable description
  ├── executor.py         # Execution module (implements SkillBase interface)
  ├── schema.json         # Input/output JSON Schema (machine-readable)
  ├── scripts/            # Helper scripts (optional)
  ├── references/         # Reference documents (optional)
  └── templates/          # Code templates (optional)
```

**Step 2: Preserve SKILL.md as the metadata carrier**

The SKILL.md file becomes a metadata document with YAML frontmatter, while actual execution logic moves to `executor.py`.

### 3.3 Executor Base Class Design

All self-sufficient Skill executors inherit from a unified base class ensuring interface consistency:

| Design Decision | Rationale |
|-----------------|----------|
| Base class auto-loads SKILL.md | L3 Skill prompts originally reside in SKILL.md; direct reuse eliminates rewriting |
| `schema.json` as separate file | Machine-readable; enables constraint layer validation and training data generation |
| `references/` and `templates/` directories | L2/L3 Skills carry domain knowledge and code templates without external dependencies |
| Provider adapter initialized per subclass | L0/L1 Skills do not need model calls; imported lazily |

### 3.4 L3 Skill Executor Implementation Pattern

L3 Skill executors follow a unified pattern: **load prompt → assemble context → invoke provider adapter → parse output**.

```
Pseudocode pattern:
  1. Construct system prompt = Skill's built-in prompt + reference files + templates
  2. Invoke a model through the provider adapter when needed
  3. Parse output into standardized file list
  4. Return SkillResult with success flag and structured data
```

## 4. Why Routing Is the Right Job for Small Models

### 4.1 The Economic Argument

The execution layer exists to separate concerns: management-layer models handle the cognitive work of planning and decomposition, while local models handle the mechanical work of routing. This separation yields:

| Benefit | Mechanism |
|---------|-----------|
| **Reduced cloud API costs** | Routing decisions are high-frequency, low-value token consumers. Moving them locally can reduce routing-token consumption at scale. |
| **Lower latency** | Local routing completes in <100ms versus 1-3s for cloud routing. For systems executing thousands of Skill calls daily, this compounds significantly. |
| **Offline capability** | L0/L1 Skill routing works without network access. Systems remain partially operational during outages. |
| **Privacy preservation** | Task descriptions can remain local for routing decisions. Sensitive business logic and proprietary code patterns are exposed to external services only when a Skill explicitly delegates work. |

### 4.2 What Routing Actually Requires

Skill routing is not "trivial"—it requires two non-trivial capabilities:

1. **Skill Matching**: Given a task description, select the most appropriate Skill from 60+ candidates. This is a text classification problem requiring understanding of both the task and each Skill's capability scope.

2. **Parameter Extraction**: Given a task description and selected Skill, extract the parameters required by the Skill's input schema. This requires understanding both natural language and structured schema constraints.

These capabilities are within reach for fine-tuned 1B-3B models when combined with constraint layers that validate and correct outputs. They do not require code generation, architectural reasoning, or complex synthesis—all capabilities that remain unreliable in small models.

### 4.3 Context Window Mitigation

A practical challenge: 60+ Skill descriptions total 4000-6000 tokens in system prompts. Combined with task descriptions and AES parameters, total context can exceed small model context windows (e.g., 4096 tokens for Llama-3.2-1B).

Mitigation strategies:

| Strategy | Description | Token Savings |
|----------|-------------|---------------|
| **Hierarchical filtering** | Send only Skill subsets (5-10) relevant to the AES node's `skill_binding.primary_skill` | ~70% |
| **Keyword indexing** | Send only Skill name + keywords, omitting detailed descriptions | ~50% |
| **Two-step matching** | Step 1: keyword filter to 5-10 candidates using deterministic code; Step 2: local model precise matching from candidates | ~70% |
| **RAG retrieval** | Vectorize Skill descriptions; retrieve top-K matches for task description | ~80% |
| **Upgrade model** | Use models with larger context windows (e.g., Qwen2.5-3B with 32768 tokens) | N/A |

**Recommended approach**: Two-step matching. The first step filters to 5-10 candidates via deterministic keyword matching. The second step sends the candidate list and task description to a local model for precise matching.

## 5. AES Granularity and Skill-Callable Tasks

### 5.1 The Granularity Principle

Each AES node must map to a single Skill call. If an AES node requires multiple Skills, it belongs at a higher layer (management layer) where an agent can orchestrate multiple Skill calls sequentially or in parallel.

**Atomic AES Rule**: An AES node is atomic if and only if it calls exactly one Skill.

### 5.2 AES Refinement Example

Original AES node (too coarse for a small model):

```
EX-02: Core implementation task
  Requires: substantial async code generation, reference to runtime interfaces,
            composition pattern implementation
  Small model capable: ~40% (parsing, validation, checking)
  Small model incapable: ~60% (design, coding)
```

Refined AES nodes (small model capable):

```
EX-02a: Read reference material, extract interface definitions
  → Skill: code-analyzer (L1)

EX-02b: Generate implementation skeleton
  → Skill: python-codegen (L3, provider-adapter delegation)

EX-02c: Implement task execution method
  → Skill: python-codegen (L3, provider-adapter delegation)

EX-02d: Run required validation commands
  → Skill: test-runner (L0)

EX-02e: Check file attributes and code standards
  → Skill: code-reviewer (L0/L3 hybrid)
```

Each refined node maps to exactly one Skill. Small models can reliably execute all five nodes.

### 5.3 AES Metadata for Routing

Each AES node includes metadata enabling reliable routing:

```yaml
node_id: "EX-02a"
node_name: "Extract interface definitions"
skill_binding:
  primary_skill: "code-analyzer"
  fallback_skill: "generic-file-reader"
input_params:
  reference_file: "runtime/interface_reference.py"
  extract_types: ["interfaces", "type_aliases"]
validation_commands:
  - "test -f {output_path}"
  - "wc -l {output_path} | awk '{print $1}' | test $1 -gt 10"
```

The `skill_binding.primary_skill` field provides the default routing target. The execution agent can use this as a shortcut (skip matching, use directly) or validation target (verify matching result matches binding).

## 6. Validation, Failure Bubbling, and Governance

### 6.1 Validation at the Skill Layer

Results should be validated at the Skill layer before returning to the execution agent. The Skill knows its output schema and can verify structural correctness before returning.

```
Validation sequence:
  1. Skill executes and produces raw output
  2. Skill validates output against output_schema
  3. If invalid: Skill retries or returns structured error
  4. If valid: Skill returns SkillResult with success=True
  5. Execution agent receives validated result
```

### 6.2 Failure Bubbling

When a Skill fails, structured error context bubbles up to the execution agent and potentially to the management layer:

```yaml
SkillResult:
  success: false
  skill_name: "frontend-design"
  error:
    code: "LLM_INVOCATION_FAILED"
    message: "Provider adapter returned status 429: Rate limit exceeded"
    retryable: true
    fallback_suggested: "Use cached template-based fallback"
  duration_ms: 2341
```

The error structure enables:
- **Execution agent**: Retry logic, fallback Skill selection
- **Management layer**: Decision on whether to retry, wait, or escalate to human

### 6.3 Constraint Alignment

This architecture aligns with the Constraint Hierarchy Alignment Framework (document 3.9.11), which addresses "tool path bypass behavior." The mapping:

| 3.9.11 Governance Layer | Original Target | Execution Layer Implementation |
|------------------------|-----------------|-------------------------------|
| **Soft constraints** | Prompt reinforcement for tool path selection | System prompts listing 60+ Skills with invocation formats |
| **Hard constraints** | Logit masking, response prefilling, interception hooks | Skill whitelist validation, parameter schema validation, format extractors |
| **Fine-tuning** | LoRA/QLoRA for tool-call preference learning | LoRA fine-tuning for Skill matching and parameter extraction |

**Critical distinction**: the constraint-alignment problem addresses capable models that choose not to call tools, while this architecture addresses weak models that cannot invoke tools reliably. Both use combinations of constraints and fine-tuning, but the problem type differs:

- Constraint alignment targets frontier models that have capability but may choose to bypass tools
- This architecture's governance targets local weak models (1B-3B) that lack capability to bypass, but also lack capability to invoke correctly

This yields fundamentally different governance strategies: constraint alignment restrains capable models, while this architecture enables weaker models. Skill self-sufficiency is the key enablement mechanism: capabilities the execution-layer model need not possess are encapsulated within Skills.

## 7. Engineering Example: Pipeline Execution

### 7.1 Pipeline Execution Skill

A pipeline execution Skill demonstrates the execution-layer architecture in practice. It encapsulates:

- **Pipeline execution engine**: Implements complete execution loops, state serialization, and checkpoint mechanisms
- **Error diagnosis and auto-repair**: Defines recovery flows: S1-S4 error classification, LLM auto-fix, retry backoff strategies
- **Multi-stage task orchestration**: DAG validation logic defines node execution order
- **Test execution and quality review**: Required validation command execution, review prompt construction, and standardized review calls

### 7.2 Self-Healing Pipeline

When a pipeline encounters errors, the system executes a self-healing flow:

```
1. Anomaly Detection
   └→ Read pipeline_state.json
       └→ Identify failed/paused nodes

2. Context Capture
   ├→ Read AES YAML (deliverables, constraints, acceptance criteria)
   ├→ Extract error information (FAIL_FILES, fail_reason)
   └→ Read related source file content (first 2000 chars)

3. LLM Repair
   ├→ Build structured repair prompt
   ├→ Invoke the configured repair provider
   └→ Repair provider generates and applies fix

4. State Reset
   ├→ node.state → pending
   ├→ retries → 0
   └→ error → null

5. Pipeline Resume
   └→ Execute --resume
```

### 7.3 Constraint Auto-Adjustment

A practical pattern: when generated files exceed `max_lines_per_file` constraints, the system auto-relaxes rather than forcing breakdown:

```
Algorithm:
  violations = check_file_line_limits(stage, state)
  for violation in violations:
    actual_lines = violation["actual_lines"]
    new_limit = int(actual_lines * 1.2)  # Auto-relax by 20%
    adjust_max_lines_in_yaml(violation["aes_file"], new_limit)
```

This embodies the "constraints first, dynamic adjustment" principle—constraints remain traceable while adapting to actual model outputs.

## 8. Relationship to EAPF, DCHC, and Constraint Alignment

### 8.1 DCHC Framework Positioning

This work extends the DCHC (Divide-and-Conquer Human-AI Collaboration) framework's L1 execution layer:

| Layer | Document | Responsibilities | Model Requirements |
|-------|----------|------------------|-------------------|
| L3: Human-AI Collaboration | DCHC | Human creativity → Architecture documents → Stage planning | Frontier/domain model |
| L2: Planning Refinement | Related architecture work | Architecture → Stage → AES | Frontier/domain model |
| **L1: Automatic Execution** | **This paper** | **AES → Skill call → Result validation** | **Local small model** |

### 8.2 Skill as Prompt vs. Skill as Executable Component

A critical distinction this paper clarifies:

- **Skill as prompt**: The Skill is a text file providing guidance to the calling agent, which performs the actual work. Quality depends on caller capability.

- **Skill as executable component**: The Skill is a self-contained executable unit with runtimes, validators, and optional model delegation. Quality is self-contained.

The architectural goal is converting Skills from "prompt" to "executable component" paradigm without disrupting existing Skill interfaces.

### 8.3 Integration with Constraint Alignment Framework

The related constraint-alignment framework provides governance methodology for the execution layer:

- **Soft constraints**: System prompts enumerate 60+ Skills with invocation formats, guiding model behavior
- **Hard constraints**: Schema validation, whitelist checking, and format extraction ensure outputs conform to interface contracts
- **Fine-tuning**: LoRA adapters train Skill matching and parameter extraction capabilities directly into the model

## 9. Limitations and Open Questions

### 9.1 Current Limitations

1. **LLM repair success rate depends on model capability**: Complex logical errors may resist automated repair attempts. The self-healing pipeline degrades gracefully but may require management layer intervention for sophisticated failures.

2. **Context window constraints**: Small models with limited context windows require mitigation strategies such as hierarchical filtering and two-step matching, which add complexity.

3. **Skill registry maintenance**: As Skill libraries grow to 60+ entries, maintaining accurate metadata (keywords, task types, exclusion rules) becomes a maintenance burden.

4. **Cross-Skill state passing often relies on the filesystem**: Many multi-Skill workflows use file-based state transfer. More explicit state-sharing contracts may be needed for complex workflows.

5. **Fine-tuning data generation**: Creating training data for Skill matching requires annotating large numbers of (task description, Skill) pairs—a significant investment.

### 9.2 Open Questions

1. **Optimal AES granularity**: What is the right size for AES nodes? Too fine creates overhead; too coarse exceeds small model capability. Systematic empirical evaluation needed.

2. **Skill composition language**: Can we define a declarative DSL for multi-Skill workflows that executes reliably without requiring full management layer intervention?

3. **Adaptive Skill selection**: Rather than fixed routing, can execution agents learn from success/failure histories to improve Skill selection over time?

4. **Skill versioning and compatibility**: How do we handle Skill updates without breaking existing AES node bindings?

5. **Multi-tenant Skill isolation**: In enterprise deployments, can Skills from different tenants be safely composed without information leakage?
