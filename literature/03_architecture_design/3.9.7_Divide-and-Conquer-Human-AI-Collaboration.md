# Divide-and-Conquer Human-AI Collaboration: A Layered Automation Model for Multi-Agent Systems

## Abstract

This paper presents the **Divide-and-Conquer Human-AI Collaboration (DCHC)** framework, which addresses two core challenges in multi-agent systems: **role explosion** and **black-box automation**. Building on DACES/HIL-5/MCP theory, the framework divides complex workflows into three distinct layers: a **human-driven creative decision layer**, an **AI-driven structural decomposition layer**, and an **AI-driven automated execution layer**. By introducing **evolvable skills** as cross-layer capability units and combining them with **threshold-adjustable quality control loops**, the framework enables humans to retain creative direction and responsibility while empowering AI to handle structured, verifiable execution. The framework is illustrated through a generalized "architecture design -> stage planning -> AES implementation" workflow, detailing the philosophical foundations, theoretical model, implementation methods, and generalizability implications.

## Agent Reading Guide

### Core Claim

Effective human-AI collaboration should divide work by comparative advantage: humans retain creative direction and responsibility, while agents handle structured decomposition and verifiable execution.

### Key Definitions

| Term | Definition |
|------|------------|
| **Agent** | An autonomous entity capable of perceiving, reasoning, and acting within a system. |
| **Skill** | A self-contained, evolvable capability package that encapsulates specific workflows, domain knowledge, and automation scripts. |
| **Harness** | The orchestration layer that coordinates agents, skills, and resources within a multi-agent system. |
| **Execution Asset** | A machine-readable specification (e.g., AES) that defines inputs, outputs, constraints, and acceptance criteria for a task. |
| **Promotion** | The mechanism by which outputs from one layer become validated inputs for the next layer in the automation hierarchy. |
| **L3 Layer** | Human-driven creative decision layer responsible for high-level architecture and strategic direction. |
| **L2 Layer** | AI-driven structural decomposition layer that breaks high-level designs into executable stages. |
| **L1 Layer** | AI-driven automated execution layer that implements and validates individual tasks. |
| **Quality Gate** | A threshold-based validation checkpoint that ensures outputs meet confidence criteria before promotion. |

### Do Not Misread This As

- **Do not** interpret this as a fully autonomous replacement for human judgment. DCHC explicitly reserves creative decision-making and responsibility for humans.
- **Do not** view the three layers as isolated components. They form a tightly integrated pipeline with structured handoffs and feedback loops.
- **Do not** assume skills replace agents. Skills are capability packages that agents invoke; they do not possess autonomous agency.
- **Do not** treat quality thresholds as fixed. They are user-configurable based on project context, risk tolerance, and stage of development.
- **Do not** equate "human-in-the-loop" with "human as passive reviewer." In DCHC, humans are active creative directors, not merely approvers.

### Reusable Rules

1. **Human Retains**:
   - High-level architecture and strategic direction
   - Creative synthesis and novel concept generation
   - Final responsibility for decisions and outcomes
   - Boundary setting for automation depth

2. **AI Handles**:
   - Structured decomposition of designs into executable stages
   - Rule-based task planning and dependency resolution
   - Repetitive, verifiable execution of defined tasks
   - Automated testing, validation, and code review

3. **Skill Design**:
   - Encapsulate workflows in self-contained packages with SKILL.md, scripts/, references/, and assets/
   - Support dynamic discovery and invocation by agents
   - Include evolution mechanisms based on execution feedback

4. **Quality Control**:
   - Use configurable confidence thresholds (0.7-0.9) for quality gates
   - Allow manual adjustment based on project context
   - Implement automated retry and escalation for failed validations

5. **Handoff Structure**:
   - Use structured or semi-structured data formats at human-AI boundaries
   - Define clear contracts with input/output schemas
   - Separate human responsibility for correctness from AI responsibility for rule-based processing

## Problem: Role Explosion and Black-Box Automation

As large language models (LLMs) and agent technologies advance rapidly, building multi-agent systems capable of autonomous complex task execution has emerged as a research and application frontier. However, the pursuit of high automation reveals two growing contradictions:

### Role Explosion: Overspecialization of Agent Responsibilities

A common pattern in current multi-agent architectures is designing independent agents for each specific subtask or capability domain. For example, in a software project automation system, one might find separate "Product Manager Agent," "Architect Agent," "Frontend Developer Agent," "Backend Developer Agent," "Test Agent," and "Deployment Agent." While this design offers clear responsibilities, it introduces significant side effects:

- **Escalated coordination overhead**: Agents require complex communication protocols for task handoffs and state synchronization, with substantial time consumed by coordination rather than execution.
- **Increased system fragility**: Failure or bottlenecks in any single agent can block the entire workflow; system reliability depends on the weakest link.
- **Fragmented knowledge**: Domain knowledge is distributed across agents, preventing formation of unified, evolvable knowledge systems.
- **High development and maintenance costs**: Each new task type may require designing, training, and integrating a new agent.

### Black-Box Automation: Loss of Human Control

The opposite extreme pursues end-to-end full automation. The system receives high-level goals (e.g., "build an e-commerce website") and attempts to autonomously complete requirements analysis, design, coding, testing, and deployment. While seemingly efficient, this approach harbors significant risks:

- **Blurred creative and responsibility ownership**: Who bears final responsibility for design decisions? When ethical, legal, or business logic errors occur, accountability becomes difficult.
- **Uncontrollable key decision quality**: Lacking deep domain knowledge and business context, AI may make architectural decisions that appear reasonable on the surface but are fundamentally flawed or contain serious vulnerabilities.
- **Difficult debugging and correction**: When automation fails, the lack of transparency and human-understandable checkpoints makes locating root causes and implementing fixes exceptionally challenging.
- **Difficulty establishing user trust**: Users naturally maintain wariness toward fully unintervenable and incomprehensible "black-box" processes, hindering technology adoption.

## Philosophical Basis: Human as Creative and Responsibility Anchor

The philosophical foundation of DCHC acknowledges and leverages the fundamental capability differences between humans and current AI (particularly large language models), rather than attempting to make one mimic or replace the other. This is not a compromise but an active division-of-labor strategy based on **comparative advantage**.

### Comparative Advantages

| Human Strengths | Current AI Strengths |
|-----------------|---------------------|
| **Abstraction and creativity**: Extract novel concepts, visions, and high-level architectures from vague requirements | **Massive pattern recognition and generation**: Rapidly process massive structured or semi-structured information and generate pattern-compliant outputs |
| **Value judgment and responsibility**: Understand ethical, commercial, and social contexts and accept final responsibility for decisions | **Relentless repetitive execution**: Execute clearly defined repetitive tasks with extremely high consistency |
| **Deep domain knowledge**: Possess years of accumulated, difficult-to-encode tacit knowledge | **Rapid iteration and exploration**: Generate numerous solution variants in short periods for human evaluation |
| **Big-picture thinking and cross-domain association**: Connect seemingly unrelated domains for breakthrough innovation | **Strict rule adherence**: Precisely execute predetermined logical constraints and format requirements |

DCHC advocates reserving workflow stages **most suitable for humans** for humans and delegating stages **most suitable for AI** to AI. Specifically, humans should dominate the **two ends** of the workflow: **initial creative input** and **final decision approval**. AI serves as the intermediate **translator and executor**, converting human high-level intent into actionable specific steps and reliably executing them.

### Human as Creative and Responsibility Anchor

In the DCHC framework, the human role is redefined as the system's **creative and responsibility anchor**, manifested in two critical stages:

1. **Master design generation and deep review**: Top-level system architecture, core business logic, and key non-functional requirements must be finalized by humans or through explicit human review. This process is not one-time but iterative; humans can call different AI models as "consultants" for multi-perspective review, but final decision authority remains with humans. This ensures **directional correctness** and **clear accountability**.

2. **Automation boundary setting and adjustment**: Humans have the authority to decide which tasks to delegate to AI for automation and the depth of automation (e.g., review threshold configuration). This boundary is not fixed but dynamically adjustable based on task importance, risk level, and AI maturity.

### AI as Structuring and Execution Engine

The AI role becomes a **high-fidelity amplifier** of human intent. It does not attempt to understand the "why" behind intent (which belongs to human creative and value domains) but focuses on "how to achieve it":

1. **Structured decomposition**: Breaking down human-defined, relatively macro-level master designs into fine-grained, executable stages or tasks according to explicit rules (dependency relationships, workload balancing, architectural principles). This process is deterministic and verifiable.

2. **Specification generation**: Generating machine-readable, unambiguous execution specifications (e.g., Agent-Executable Specification, AES) for each decomposed task. The specification strictly defines inputs, outputs, constraint conditions, and acceptance criteria.

3. **Reliable execution and validation**: Invoking appropriate tools or large models to strictly execute tasks according to specifications and automatically running tests, conducting code reviews, and other verification activities to ensure outputs meet expectations.

Through this division of labor, the DCHC framework aims to achieve the ideal collaborative state of **"human at the helm, AI at the oars"**. Humans always control the course and final decision authority, while AI provides powerful, precise, and tireless propulsion.

## Three-Layer Automation Model

The DCHC framework builds on DACES, HIL-5, and MCP (Memory-Context-Planning) with key extensions. The core is the **three-layer automation model**, with each layer having clear human-AI division of labor, input-output specifications, and quality control mechanisms.

### L3: Human-Driven Creative Decision Layer

- **Primary driver**: Human experts
- **Core inputs**: Vague business requirements, market insights, technical vision
- **Core activities**:
  1. **Master architecture design**: Developing system-level architecture diagrams, core components, key interaction flows, and non-functional requirements
  2. **Multi-round deep review**: Using AI as a "consultant panel," calling different models for multi-perspective design review while retaining final decision authority
  3. **Responsibility confirmation**: Confirming the final design document and explicitly designating it as the automation pipeline input source
- **Output artifacts**: A master architecture design document with structured data appendices (component manifests, work estimates) serving as the human-machine handoff bridge
- **Quality control**: Depends on human professional judgment and multi-model cross-validation

### L2: AI-Driven Structural Decomposition Layer

- **Primary driver**: AI decomposition skill
- **Core inputs**: Master design documents from L3 layer containing structured data
- **Core activities**:
  1. **Parsing and extraction**: Automatically parsing documents to extract structured information including component mappings, work estimates, and dependency relationships
  2. **Rule-based splitting**: Decomposing master designs into an ordered, appropriately granular sequence of stages based on predefined algorithms (component clustering, workload balancing, dependency topological sorting)
  3. **Planning document generation**: Generating detailed planning documents for each stage, including objectives, component lists, implementation plans, and acceptance criteria
- **Output artifacts**: A set of standards-compliant **stage planning documents**
- **Quality control**: **Embedded review**. After splitting completion, automatically calling the system's review component to conduct architecture review on generated planning; only stage plans reaching preset confidence thresholds (e.g., 0.8) are passed to the next layer

### L1: AI-Driven Automated Execution Layer

- **Primary driver**: AI execution skill
- **Core inputs**: Stage planning documents from L2 layer that passed review
- **Core activities**:
  1. **AES generation**: Based on stage planning, calling large models to generate specific, executable Agent-Executable Specification (AES) node files
  2. **Large model scheduling and execution**: Assigning appropriate large models to each AES node to execute coding, testing, and other specific tasks
  3. **Automated validation**: Automatically running unit tests, integration tests, and conducting code style and security reviews
  4. **Cyclic advancement**: After all nodes in the current stage pass validation, automatically triggering the next stage's workflow until all stages complete
- **Output artifacts**: **Executable codebase**, **test reports**, **review records**, and final **project completion status**
- **Quality control**: Automated checkpoints at each stage (AES validation, test passing, review approval). Any checkpoint failure triggers retry or error handling workflows

## Skills as Evolvable Capability Units

Skills are the core abstraction encapsulating specific capabilities (especially for L2 and L1 layers) in the DCHC framework. Their design draws on established agent-tooling practices and emphasizes **evolvability**:

- **Standard structure**: A complete skill contains `SKILL.md` (description and workflow), `scripts/` (automation scripts), `references/` (domain knowledge), and `assets/` (resources), forming a self-contained capability package.
- **Dynamic loading and discovery**: Skills are registered through the system's skill loader, enabling agents to dynamically discover, match, and invoke them.
- **Evolution mechanism**: Skills possess self-optimization capabilities based on execution feedback. The system records skill execution successes and failures, periodically analyzes patterns, and automatically generates optimization suggestions (patches) for skill descriptions, scripts, or reference documents, which are applied after human review, making skills "smarter with use."

In DCHC, decomposition skills and execution skills encapsulate complex workflow automation capabilities into reusable, iterable components.

## Adjustable Quality Gates

Quality control is the core thread through all three layers. The DCHC framework introduces **"threshold-adjustable quality control loops"** to balance efficiency and quality:

- **Review confidence**: The system conducts automated review on any intermediate AI-generated artifacts (designs, plans, code) and outputs a confidence score (0.0-1.0).
- **User-configurable tiers**: The framework provides multiple predefined threshold tiers (e.g., **lenient(0.7)**, **standard(0.8)**, **strict(0.9)**), allowing users to select appropriate tiers based on project stage, risk tolerance, or quality requirements. This grants users flexible control authority.
- **Automated gates**: Only when intermediate artifact review confidence is **greater than or equal to** the currently configured threshold does the workflow proceed to the next stage. Otherwise, it triggers a remediation workflow (automatic retry or escalation to humans).
- **Value**: Avoids "one-size-fits-all" quality standards, enabling rapid iteration during exploration stages and pursuing high quality during release stages, achieving quality control that is **elastic** and **humanized**.

## Structured Handoff Artifacts

To ensure smooth human-AI collaboration, the DCHC framework emphasizes using **structured or semi-structured data formats** as "contracts" at key handoff points. The most important handoff occurs between L3 and L2:

- **Agreed format**: When humans write master design documents, they **must** include structured data appendices in specific formats (e.g., Markdown tables), such as:
  - **Component manifest tables**: Explicitly listing each component, its belonging layer, assigned stage, and priority
  - **Work estimate tables**: Explicitly listing base hours and buffer hours for each stage
- **Machine readability**: These table formats are relatively fixed, facilitating accurate extraction by L2 layer skills through parsers, avoiding natural language understanding ambiguity.
- **Responsibility separation**: Humans are responsible for ensuring the **correctness** of these structured data (whether component assignment is reasonable, whether work estimates are accurate), while AI is responsible for ensuring the **splitting process** based on this data complies with established rules. This clearly divides human-AI responsibility boundaries.

Through the combination of these four components, the DCHC framework constructs a theoretical model that allows deep human control while fully unleashing AI automation potential, with both elasticity and evolution capabilities.

## Generalized Workflow Example

The following generalized workflow illustrates how DCHC operates across domains beyond software development:

```
[L3: Human Layer]
Human defines high-level objectives and creative direction
        ↓ Output: Master design with structured data

[L2: AI Decomposition Layer]
Skill parses and splits design into executable stages
        ↓ Output: Stage planning documents

[L1: AI Execution Layer]
Skill generates specifications, executes tasks, validates results
        ↓ Output: Completed deliverables + reports
```

### Example Domain: Data Analysis Workflow

1. **L3**: Human defines analysis goals, hypotheses, and data sources
2. **L2**: Skill decomposes into data collection stages, transformation stages, and analysis tasks
3. **L1**: Skill executes data extraction, cleaning, analysis, and validation, generating reports

### Example Domain: Documentation Generation

1. **L3**: Human defines knowledge framework and content strategy
2. **L2**: Skill splits into chapters, sections, and individual content units
3. **L1**: Skill generates content, performs consistency checks, and compiles final documents

## Relationship to DACES/HIL-5/MCP and AES

DCHC framework integrates with and extends existing theoretical foundations:

### DACES Integration

DACES defines a dual-state control framework for moving between stable execution and adaptive evolution, but it does not by itself specify human-AI collaboration layer boundaries. DCHC's three-layer model provides specific automation layering guidance for DACES implementations.

### HIL-5 Integration

HIL-5 emphasizes deep human-in-the-loop participation but lacks systematic analysis of "which tasks should be reserved for humans and which can be safely delegated." DCHC's tripartite division of "creative decision vs. structured decomposition vs. automated execution" provides an actionable framework for HIL-5 practice.

### MCP Integration

MCP organizes Memory, Context, and Planning across agent sessions and task execution. DCHC maps this three-axis model onto collaboration layers: human-authored design becomes governed memory, L2 decomposition controls planning, and L1 execution receives bounded context for verifiable work.

### AES Integration

The Agent-Executable Specification (AES) serves as the primary machine-readable artifact in the L1 layer. AES nodes define the contracts for individual tasks, including inputs, outputs, constraints, and validation criteria. DCHC explains how AES nodes are generated from L2 stage plans and systematically executed and validated.

### Relationship to EAPF and Harness Engineering

DCHC defines the collaboration topology for human-agent work. EAPF defines how intermediate outputs become reusable execution assets, and Harness Engineering defines the control environment that keeps those assets constrained, verified, and correctable. In this sense, DCHC is the division-of-labor layer: it specifies who creates intent, who decomposes intent, and who executes verifiable units of work.

## Engineering Implications

### 1. Skill Architecture Over Agent Proliferation

**Do not create independent agents for every task.** Prioritize encapsulating capabilities as reusable, evolvable skills invoked by existing agents as needed. This prevents role explosion while maintaining system modularity.

### 2. Human Control at Decision Layers

**Retain human control at critical decision layers**, especially for decisions involving creativity, responsibility, and security. Do not fully delegate these to AI.

### 3. Structured Data at Handoff Points

**Use structured data formats at human-AI handoff points.** This is more reliable than natural language handoffs and forms the foundation of automated workflows.

### 4. User-Controlled Automation Depth

**Let users control automation depth.** Different projects and stages require different quality-efficiency balance points. Configurable quality gates provide this flexibility.

### 5. Skill Evolution Loops

**Implement closed-loop skill evolution**. Collect execution feedback, analyze patterns, generate optimization patches, and apply them after human review. This ensures skills improve over time rather than degrading.

### 6. Unified LLM Invocation Paths

**Unify all LLM invocations through system adapters**, not through external CLI or independent channels. This enables monitoring, logging, routing, and consistent behavior across all AI interactions.

## Limitations and Open Questions

### Current Limitations

1. **Human bottleneck at L3 layer**: While automation reduces human involvement in decomposition and execution, L3 layer human involvement remains substantial. For projects requiring rapid iteration, the human design stage may become a bottleneck.

2. **Skill discovery challenges**: As skill ecosystems grow, agents may struggle to discover and select appropriate skills, leading to either skill underutilization or redundant skill creation.

3. **Quality threshold calibration**: Determining optimal confidence thresholds (0.7, 0.8, 0.9) requires domain-specific experimentation and may need adjustment based on project maturity.

4. **Dependency on structured data**: The framework's reliability depends on humans providing accurate structured data in design documents. Incorrect or incomplete structured data can cascade into downstream failures.

### Open Research Questions

1. **Automatic skill synthesis**: Can the system automatically synthesize new skills from successful subagent explorations without human intervention? What safety boundaries are needed?

2. **Dynamic threshold adjustment**: Can confidence thresholds be automatically adjusted based on project context, historical performance, and real-time feedback?

3. **Multi-human collaboration**: How does DCHC extend to scenarios with multiple human stakeholders with potentially conflicting creative visions? How are conflicts resolved?

4. **Cross-organization skill sharing**: What governance models, licensing frameworks, and trust mechanisms enable secure skill sharing between organizations?

5. **Measuring "creative contribution"**: How do we quantitatively measure human creative input versus AI execution output for attribution and evaluation purposes?

6. **Failure mode taxonomy**: What is a comprehensive taxonomy of failure modes in DCHC workflows, and what are appropriate automated recovery strategies for each?
