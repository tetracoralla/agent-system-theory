# Claude Skills智能体架构：基于技能的AI能力扩展新范式

## 摘要

本文深入分析Claude Skills智能体架构的理论原理与技术实现，揭示其作为"AI的Unix时刻"的创新价值。Claude Skills代表了从静态提示词工程到动态技能工程的范式转变，通过"渐进式披露"架构和"元工具"设计，实现了对传统AI能力扩展方式的根本性重构。与Google Gemini 3等竞品相比，Claude Skills采用模块化、文件系统导向的架构，将领域专业知识封装为可动态加载的Markdown技能包，通过双通道通信机制和BDI推理循环，在保持高效上下文管理的同时提供强大的推理与执行能力。**Claude Skills的核心优势在于其"轻量加载"机制，使得即使面对包含大量技能的复杂系统，模型的响应速度仅下降不到5%，Token利用率显著提升**，为构建可组合、可扩展的AI智能体提供了革命性框架。

---

## 1. 概念重构：从提示词工程到技能工程

### 1.1 核心定义的澄清

随着大型语言模型（LLM）从单纯的文本生成工具向自主智能体演进，AI系统的架构正在经历一场根本性的重构。在这一转型过程中，几个关键概念需要明确：

- **智能体（Agent）**：指具备自主推理和决策能力的系统，如基于Claude 3.5 Sonnet或Gemini 3构建的AI助手，是执行者，拥有核心推理能力。

- **技能（Skill）**：Claude Skills是智能体可动态加载的"能力模块"，**不是智能体本身，而是智能体可调用的"能力包"**，包含领域专业知识、指令集和执行逻辑。

- **工具（Tools）**：传统工具指可直接执行的操作（如运行代码、读取文件），通常通过API调用实现，而Claude Skills引入了"元工具"（Meta Tools）的概念，用于加载和管理其他技能。

这种概念重构反映了AI开发从**提示词工程**（Prompt Engineering）向**技能工程**（Skill Engineering）的范式转变。传统提示词工程需要开发者手动编写和维护长篇提示词，将所有可能需要的知识一次性加载到上下文中，随着功能增加，提示词膨胀导致上下文窗口（Context Window）被迅速占满，造成"上下文稀释"（Context Dilution）和"token爆炸"（Token Explosion）问题。

### 1.2 从静态提示到动态技能的范式转变

Claude Skills的核心价值在于引入了"动态技能加载"的概念，解决了传统AI系统面临的三大核心挑战：

- **认知过载**：传统系统要求模型一次性处理大量无关信息，而Claude Skills通过"渐进式披露"（Progressive Disclosure）机制，**仅在需要时加载相关技能内容**，保持模型注意力的聚焦。

- **上下文浪费**：Claude Skills采用三级渐进式加载策略，将技能内容划分为元数据、核心指令和资源脚本，**根据任务需求按需加载不同层级内容**，最大限度降低token消耗。

- **能力扩展性**：传统提示词工程需要修改基础提示词以添加新功能，而Claude Skills采用模块化设计，**通过增减独立技能目录即可扩展能力**，无需修改核心系统逻辑。

这种范式转变类似于人类获取知识的方式：先知道"有什么工具"，再根据任务选择"合适的工具"，最后"使用工具的必要部分"完成工作。Claude Skills使模型行为更接近人类信息处理方式，实现了从"全局知识"到"精准知识"的转变。

## 2. 技术架构：Claude Skills的三重机制

### 2.1 Skill工具与元工具架构

Claude Skills的技术架构建立在三个关键组件之上：

**第一，Skill工具（元工具）**：在Claude的工具数组（Tools Array）中，存在一个特殊的"Skill"元工具，其描述字段包含了所有可用技能的名称、简短描述和触发条件。每个技能（如pdf、code审查、邮件撰写）本质上是一段待注入的提示词模板，存储在独立的目录中。

**第二，双通道通信机制**：Claude Skills采用创新的双通道通信设计，解决了"既要向用户展示进度，又要向AI注入大量指令而不刷屏"的矛盾：

- **显性通道**：向用户发送简短的元数据消息，如"正在加载PDF技能..."，让用户了解系统状态。

- **隐形通道**：通过isMeta: true标记向Claude注入包含详细技能指令的消息，这些内容仅存在于对话上下文中，用户不可见。这种隐形通道可携带数千字的详细指令，是真正指导Claude工作的核心。

**第三，动态上下文注入原理**：当Claude决定调用某个Skill时，系统会加载该Skill的核心Markdown文件（SKILL.md），将其展开为详细指令，作为新的用户消息注入到会话上下文，同时修改执行上下文（如允许使用的工具、模型选择等）。这种注入并非简单地追加文本，而是通过**声明式、纯提示**的方式扩展模型能力，使Claude在后续对话中换一种"大脑配置"去处理任务，而无需执行任何代码。

2.2 BDI推理循环与Skill的协同工作

Claude Agent SDK采用**BDI（Belief-Desire-Intention）推理循环**架构，与Skill系统形成紧密协同：

1. **Belief（信念）更新**：SDK通过next()函数驱动循环，首先更新Claude的信念状态。在此过程中，Claude会扫描已安装的Skill元数据，了解当前环境中可用的能力模块。这些元数据轻量级驻留于固定上下文中，不会因技能数量增加而显著膨胀。

2. **Desire（欲望）生成**：基于当前任务需求，Claude分析元数据，识别可能匹配的Skill。**这一匹配过程完全依赖Claude自身的语言理解能力**，而非硬编码路由或关键词匹配，赋予系统更高的灵活性和准确性。

3. **Intention（意图）确定**：Claude评估匹配Skill的相关性，并生成最终意图。如果意图涉及敏感操作（如文件修改、网络访问），Claude会通过显性通道向用户请求确认，实现"人在回路"（Human-in-the-loop）的安全控制。

4. **Skill激活与执行**：一旦意图确认，Claude激活对应Skill，根据任务需求按需加载不同层级内容：
   - Level 1（元数据）：初始加载，仅包含技能名称和简介，始终驻留上下文。
   - Level 2（核心指令）：当任务明确需要该技能时加载，包含技能详细说明和使用方法。
   - Level 3（资源脚本）：仅在执行过程中需要时加载，如特定错误处理脚本或资源文件。

5. **行动计划执行**：Claude基于已注入的Skill内容生成行动计划，并通过SDK提供的工具执行。执行结果会反馈到信念状态，形成闭环。

这种BDI循环与Skill的协同工作，实现了**"轻量级推理 + 精准知识注入"**的平衡，使Claude能够在保持高效推理能力的同时，动态获取和应用领域专业知识。

2.3 MCP协议与技能生态

Claude Agent SDK的核心设计理念是"给Claude一台电脑"，通过**Model Context Protocol (MCP)**协议实现与外部工具和服务的无缝集成：

- **MCP架构**：采用Client-Host-Server模型，基于JSON-RPC 2.0构建，提供标准化接口：
  - **Host**：运行LLM的应用环境（如Claude Code或桌面版）。
  - **Client**：模型侧的通信中间件，负责将模型指令转换为标准请求。
  - **Server**：外部能力提供者，封装工具和资源，通过MCP协议与模型交互。

- **技能生态**：Claude Skills形成了覆盖多种场景的技能生态：
  - **内置技能**：SDK提供18+个内置工具，覆盖代码执行、文件操作、Web搜索等核心场景。
  - **自定义技能**：开发者可通过skill-creator技能自定义Markdown格式的专家手册，包含指令、脚本与资源。
  - **市场分发**：支持通过插件市场（.claude-plugin/marketplace.json）一键安装和分发技能，实现"一次编写，多处复用"。

- **安全机制**：MCP协议内置了强大的安全保护：
  - **权限控制**：通过allowedTools/disallowedTools和permissionMode实现细粒度控制。
  - **沙箱隔离**：技能在隔离环境中运行，防止越权访问。
  - **检查点回滚**：自动保存代理进度，支持任务回滚。

这种架构使Claude能够像人类一样使用计算机工作，拥有在代码库中查找文件、编写和编辑文件、运行调试等能力，成为一个真正的通用智能体。

## 3. 具体实现：Claude Skills的模块化设计

### 3.1 技能目录结构与内容组织

一个Claude Skill本质上是一个包含特定结构的文件夹，遵循标准化的目录结构：

```bash
skills/
├── excel/
│   ├── SKILL.md         # 技能核心文档（YAML+Markdown混合格式）
│   ├── resources/
│   │   ├── template.xlsx # 模板文件
│   │   └── error Guide.md # 错误处理指南
│   ├── scripts/
│   │   ├── data parsing.py # 数据解析脚本
│   │   └── chart generation.js # 图表生成脚本
│   └── metadata.json      # 技能元数据（可选）
└── pdf/
    ├── SKILL.md
    └── ...
```

**SKILL.md是技能的核心**，采用YAML+Markdown混合格式，包含：
- 技能元数据：名称、版本、作者、依赖项等
- 触发条件：描述技能适用场景的自然语言文本
- 指令集：详细的工作流程、决策逻辑和执行步骤
- 资源引用：指向其他技能或外部资源的链接

这种结构化设计使技能既可独立存在，又可与其他技能组合使用，形成强大的能力生态系统。技能可本地存储（如~/.claude/skills/），也可上传到云端，通过Claude API或Claude Code使用，极大提升了部署灵活性。

### 3.2 渐进式披露的实现机制

Claude Skills的"渐进式披露"是其最核心的技术创新，通过三级加载策略实现高效上下文管理：

**Level 1（元数据）加载**：在代理启动时，仅加载技能的元数据层，包括名称、简介和基本描述。例如，PDF处理技能的元数据可能仅为"用于解析和诊断PDF文档内容"，占用约50 tokens。**这一层级内容常驻上下文，无论安装多少技能，元数据层开销恒定**，不会随技能数量增加而显著膨胀。

**Level 2（核心指令）触发**：当用户提出特定任务请求（如"分析这个PDF中的服务器日志"），Claude通过语义理解匹配到相关Skill后，会加载该Skill的SKILL.md核心指令部分。这部分包含详细的工作流程、决策逻辑和执行步骤，可能占用数百tokens。**只有当任务明确需要时，这部分内容才会被加载**，避免了无关指令的上下文稀释。

**Level 3（资源脚本）按需调用**：在执行过程中，当Claude需要特定资源（如PDF解析错误时需参考error Guide.md）或脚本（如data parsing.py）时，才会加载Level 3内容。**这些内容仅在必要时才进入上下文，且一旦任务完成或空闲超时（通常30分钟），会被自动释放**。

这种分层披露机制的释放规则可表示为：

```
Release(Sᵢ) = (TaskComplete(Sᵢ) ∨ (IdleTime(Sᵢ) > T_idle)) ∨ (DynamicTokens(D) > D_max×0.9)
```

其中，T_idle为技能空闲超时阈值，DynamicTokens(D)为动态区当前token占用量，D_max为上下文窗口最大容量。**当动态区接近满额时，系统会优先释放低优先级/低效用技能，确保核心推理不受无关信息干扰**。

### 3.3 上下文管理与BDI循环的结合

Claude Agent SDK的上下文管理模块与BDI循环紧密结合，实现高效的上下文优化：

1. **自动压缩与保留策略**：SDK内置上下文管理器，根据retentionPolicy（如recent/important/balanced）自动压缩旧内容，优先保留当前Skill的指令。例如，当处理Excel报表任务时，系统会自动清理与邮件撰写或代码调试相关的旧内容，确保上下文窗口高效利用。

2. **动态区资源分配**：将上下文分为固定区和动态区。固定区包含基础提示词和Skill元数据，动态区则根据任务需求加载不同层级的内容。**当动态区接近满额时，系统会触发压缩机制，优先保留当前任务最相关的上下文**。

3. **BDI循环与Skill加载的协同**：BDI循环的每一步都与Skill加载策略协同工作：
   - 在Belief更新阶段，仅加载元数据层级内容
   - 在Desire生成阶段，根据任务需求加载核心指令
   - 在Intention确定阶段，评估是否需要加载资源脚本
   - 在行动计划执行阶段，根据执行需要动态加载特定资源

4. **上下文感知的推理优化**：Skill加载后，SDK会优化Claude的推理过程，调整注意力权重，使模型更关注已注入的Skill内容，提高推理的精准度和相关性。

这种上下文管理机制使Claude能够处理复杂、多步骤、跨工具/跨格式的工作流自动化，同时保持高效的上下文利用率。

## 4. 与Gemini 3的对比分析：两种架构哲学的碰撞

Claude Skills与Google Gemini 3代表了两种截然不同的AI能力扩展架构，反映了Anthropic与Google在AI系统设计上的不同哲学：

### 4.1 能力扩展方式对比

| 特性 | Claude Skills | Gemini 3 |
|------|---------------|----------|
| **核心机制** | 动态加载Markdown技能包 | 原生多模态能力+API调用 |
| **扩展方式** | 技能目录结构，低代码配置 | 需要编程实现的API调用 |
| **能力内化程度** | 模块化外挂，能力外化 | 能力内化于模型或云端服务 |
| **上下文管理** | 三级渐进式披露，按需加载 | 全局一次性加载或分段检索 |
| **交互设计** | 双通道通信（用户可见+AI隐藏指令） | 单通道工具调用（显式API请求） |

数据来源：

**Claude Skills采用"外挂式"能力扩展**，将领域专业知识封装为独立的技能包，通过提示词注入和环境修改实现能力扩展。**Gemini 3则倾向于"内化式"能力扩展**，依赖模型自身的多模态理解和推理能力，通过API调用外部工具。

### 4.2 上下文效率与响应速度对比

Claude Skills在上下文效率方面具有显著优势：

- **Claude Skills**：采用"渐进式披露"架构，**仅在需要时加载对应层级内容**。例如，处理Excel报表任务时，系统只会加载Excel相关技能内容，而不会加载邮件撰写或代码调试的无关内容。Anthropic的测试显示，启用Skills后，模型的响应速度仅下降不到5%，几乎可以忽略。

- **Gemini 3**：虽然支持超长上下文（1M-2M tokens），但其能力扩展主要依赖API调用，**需要用户主动管理上下文**。在处理多技能任务时，可能需要一次性加载大量无关信息，导致上下文稀释和推理效率下降。

这种效率差异在实际应用中表现明显：**Claude Skills系统即使安装了大量技能，也不会显著影响性能或上下文窗口利用率**，而Gemini 3在处理复杂多技能任务时，可能面临上下文管理挑战。

### 4.3 适用场景与设计优势对比

两种架构在不同场景下各具优势：

**Claude Skills的优势场景**：
- **规则明确、需精确执行的领域任务**：如企业文档生成、代码规范检查、特定格式的数据处理等。
- **知识密集型任务**：需要大量领域专业知识支持的复杂决策任务。
- **低代码环境**：通过技能市场（.claude-plugin/marketplace.json）实现一键安装和部署，降低使用门槛。
- **"人在回路"场景**：需要用户确认或监督的敏感操作，如文件修改、网络访问等。

**Gemini 3的优势场景**：
- **多模态理解与生成任务**：如结合视频与文字的复杂分析、设计稿转代码等。
- **超长文本处理**：如处理百万级token的长文档或数小时视频内容。
- **实时API调用**：需要频繁调用外部API的场景，如实时数据查询、网络服务交互等。

### 4.4 技能开发与集成复杂度对比

**Claude Skills采用低代码开发模式**：
- 开发者可通过skill-creator技能自动生成技能目录结构和基本文档
- 技能核心是Markdown格式的专家手册，无需编程能力
- 支持通过技能市场（.claude-plugin/marketplace.json）一键分发

**Gemini 3则需要较高的开发门槛**：
- 插件开发需要编写符合MCP规范的Server提供新能力
- 工具调用需通过API Key配置和环境变量设置
- 多模态能力依赖模型自身参数，无法通过简单技能扩展

这种开发复杂度差异反映了两种架构的哲学差异：Claude Skills强调"人类中心"的低门槛使用，而Gemini 3则更倾向于"技术中心"的深度集成。

## 5. Claude Agent SDK：支撑Skill系统的基础设施

Claude Agent SDK是Claude Skills系统的基础设施，提供智能体运行的"操作系统"：

### 5.1 SDK的核心功能与组件

Claude Agent SDK包含以下核心组件：

1. **上下文管理模块**：负责自动压缩和管理上下文，根据retentionPolicy（如recent/important/balanced）优化上下文窗口使用，避免Token爆炸。

2. **工具生态系统**：提供18+个内置工具，覆盖文件操作（Read/Write）、代码执行（Bash）、Web搜索等场景，并通过MCP协议扩展外部能力。

3. **权限控制模块**：通过allowedTools/disallowedTools和permissionMode实现细粒度控制，确保敏感操作的安全性。

4. **推理循环引擎**：基于BDI模型，通过next()函数驱动多轮交互，整合Skill注入的上下文作为Belief更新，指导动态决策。

5. **执行环境**：提供沙箱隔离的执行环境，防止技能越权访问敏感数据，同时支持子代理并行执行。

6. **内存优化器**：根据contextWindow（上下文窗口大小）和compactionThreshold（压缩阈值）自动优化内存使用，确保复杂任务的流畅执行。

5.2 循环机制与推理流程

Claude Agent SDK的循环机制是BDI推理模型的实现：

1. **环境感知**：通过 belief update函数接收环境输入，包括用户消息和已加载的Skill内容。

2. **意图生成**：基于最新信念，通过 desire generation函数生成可能的行动意图，此阶段会触发Skill的语义发现和匹配。

3. **计划选择**：通过 intention selection函数评估并选择最佳行动计划，可能涉及Skill的激活和确认。

4. **行动执行**：通过 plan execution函数执行选定计划，可能触发Skill的进一步加载或资源调用。

5. **结果反馈**：将执行结果反馈到信念状态，形成闭环。

这种循环机制与Skill的渐进式披露相结合，实现了**"持续推理 + 按需学习"**的智能体工作模式，使Claude能够像人类一样在执行过程中不断学习和适应新技能。

5.3 内存管理与性能优化策略

Claude Agent SDK的内存管理采用多维度优化策略：

1. **上下文分区**：将上下文分为固定区（基础提示词和Skill元数据）和动态区（任务相关Skill内容），实现差异化管理。

2. **自动压缩**：当动态区token使用超过compactionThreshold（如80%）时，自动触发压缩机制，移除低优先级内容。

3. **保留策略**：支持三种保留策略：
   - recent：优先保留最近加载的内容
   - important：基于内容重要性评估结果保留
   - balanced：平衡考虑时间因素和重要性评估

4. **子代理并行**：通过subAgents配置支持多子代理并行工作，每个子代理可独立管理自己的上下文和技能加载策略。

5. **模型切换**：根据任务复杂度，自动或手动切换到更强模型（如从Claude切换到Opus），同时优化上下文加载策略以适应不同模型的token限制。

这些策略共同确保了Claude Skills系统在处理复杂、多步骤任务时的高效性能和稳定表现。

3. **复杂Skill组合的协调**：当多个Skill同时激活时，如何避免指令冲突和上下文混乱。**潜在解决方案**：引入Skill优先级系统，支持Skill间的依赖管理和冲突解决机制。

4. **跨平台Skill兼容性**：随着Claude集成到更多平台（如Xcode），Skill兼容性成为挑战。**潜在解决方案**：完善MCP协议规范，建立跨平台Skill验证和测试框架。
