# Execution Asset Promotion Framework: From Scripts to Multi-Agent Systems

## Abstract

This paper introduces the Execution Asset Promotion Framework (EAPF), a theoretical framework for understanding multi-agent systems as execution asset operating systems rather than merely collections of autonomous agents. The framework identifies a critical failure mode in multi-agent engineering: asset-level collapse, where systems systematically degrade complex capabilities into temporary scripts, undermining long-term system evolution.

The EAPF proposes an eight-form execution asset maturity ladder that formalizes the relationship between temporary scripts, LLM-calling scripts, pipelines, multi-script structures, skills, multi-skill workflows, agents, and multi-agent systems. Each form represents increasing governance responsibility rather than mere capability differences. The framework establishes four core principles: high-level-first, low-level-restricted, failure-explicit, and promotion-institutionalized.

By reframing multi-agent systems through the lens of asset promotion, this work provides both theoretical foundations and practical engineering guidelines for building systems that actively resist degradation and continuously evolve toward higher abstraction levels.

## Agent Reading Guide

### Core Claim

Multi-agent systems should be understood as execution asset operating systems, not merely collections of autonomous agents. The reliability of such systems depends not on individual agent intelligence, but on their ability to govern promotion along a formal execution asset maturity ladder.

### Key Definitions

- **Execution Asset**: A formal unit of work capability with defined contracts, validation, and governance properties
- **Asset-Level Collapse**: The systematic degradation of execution to the lowest available capability level despite higher assets being available
- **Execution Asset Maturity Ladder**: A sequence of execution asset forms from temporary scripts to multi-agent systems, where each step increases governance responsibility
- **Promotion**: The formal process of elevating execution assets to higher maturity levels based on usage patterns and failure modes
- **Capability-Based Permissions**: Authority model where permissions are tied to specific asset responsibilities rather than general capabilities

### Concept Map

```
Temporary Script (L0) → LLM Script (L1) → Pipeline (L2) → Multi-Script (L3)
    ↓
Skill (L4) → Multi-Skill (L5) → Agent (L6) → Multi-Agent (L7)
```

**Governance Properties Progression:**
- Level 0-3: Implementation-focused (execution only)
- Level 4-5: Capability-focused (contract + execution)
- Level 6-7: Organization-focused (state + responsibility)

### Do Not Misread This As

- A simple "skill-first" engineering guide
- An argument against temporary scripts (they have valid uses)
- A recommendation for immediate multi-agent adoption
- A theoretical framework without practical engineering implications
- An endorsement of spontaneous evolution over systematic governance

### Reusable Rules

1. **High-Level First**: Prefer the highest suitable existing asset when available
2. **Low-Level Restricted**: Limit low-level assets to clear, bounded use cases
3. **Failure Explicit**: Treat all asset bypasses as governance failures requiring analysis
4. **Promotion Institutionalized**: Establish formal mechanisms for asset elevation, not rely on inspiration

## 1. Problem: Asset-Level Collapse in Multi-Agent Systems

### 1.1 The Asset-Level Collapse Phenomenon

In multi-agent engineering practice, a recurring degradation pattern emerges: regardless of how sophisticated the upper system design may be, executing agents often tend to downgrade tasks to one-time temporary scripts. They do not naturally promote repetitive tasks to pipelines, encapsulate ambiguous tasks as LLM-calling scripts, consolidate high-frequency capabilities into skills, or spontaneously determine when single-agent or multi-agent coordination should take over.

This phenomenon, termed **asset-level collapse**, represents a fundamental challenge in multi-agent system construction. On the surface, it appears to reflect "insufficient model capability"; from a systems perspective, however, the core issue is not capability limitation but the absence of unified maturity ladders, promotion rules, permission boundaries, and explicit failure mechanisms. When systems allow agents to complete tasks through the lowest-level shortcut, pre-trained models often choose that locally optimal path.

### 1.2 Why "Expecting Self-Promotion" Typically Fails

A common but dangerous assumption is that sufficiently advanced models, provided with adequate context and detailed prompts, will eventually recognize the need to self-promote from temporary scripts to pipelines, skills, agents, and multi-agent systems. This paper concludes that such expectations rarely hold in most engineering systems for three fundamental reasons:

1. **Promotion is governance behavior, not natural reasoning behavior**: Escalating from temporary script to skill involves not merely "finding a better implementation approach" but defining contracts, encapsulating runtimes, establishing validators, creating review entry points, determining ownership, and planning future reuse. This constitutes organizational governance, not single-step reasoning.

2. **Promotion benefits are global, while models perceive local benefits**: The long-term value of creating a skill is substantial, but completing the current task through a temporary script is typically faster. Models naturally optimize for local rounds rather than long-term system asset structures.

3. **Without institutional mechanisms, promotion lacks triggers**: True promotion stems from explicit thresholds, review discoveries, reuse statistics, failure logs, and management decisions—not from occasional moments of inspiration.

The correct question therefore is not "how to make agents self-learn promotion," but:

> How to design promotion itself as a formal path enforced by the system, captured by reviews, fed back to training, and recognized by the organization.

## 2. The Execution Asset Maturity Ladder

### 2.1 The Unified Proposition

The relationship `temporary script -> LLM-calling script -> pipeline script -> multi-script -> skill -> multi-skill -> agent -> multi-agent` represents not a collection of parallel concepts but a **unified execution asset ladder** with increasing inclusion relationships. Higher levels do not negate lower levels; rather, they absorb lower levels as internal implementation mechanisms while adding new governance semantics.

The unified proposition of this framework is:

> Each higher-level execution concept contains all execution capabilities of the preceding level but additionally incorporates new contracts, state, validation, reuse, governance, and organizational semantics.

For example:
- Skills can internally contain scripts, LLM calls, pipelines, templates, and multi-script combinations
- Agents can call one or more skills while maintaining their own state machines, review logic, and retry mechanisms
- Multi-agent systems can allow multiple agents to each call skills, pipelines, and scripts

The differences between these levels therefore lie not in "who is more capable" but in "who assumes higher-level system responsibilities."

### 2.2 The Eight-Form Execution Asset Model

**Table 2-1: Eight-Form Execution Asset Model**

| Level | Name | New Content | Minimum Contract | Typical Use Cases | Should Not Handle |
|-------|------|------------|-----------------|------------------|------------------|
| **L0** | Temporary Script | One-time deterministic implementation | Input/output via context | One-time, low-risk, clearly non-reusable tasks | High-frequency tasks, ambiguous tasks, cross-project tasks |
| **L1** | LLM Script | Fuzzy transformation capability in scripts | Prompt template + input/output constraints | Single-step fuzzy processing (summarization, transformation, generation) | Multi-step orchestration, long-term state management |
| **L2** | Pipeline Script | Multiple steps in stable sequence | Step order + state checkpoints | Repetitive fixed step chains | Cross-domain reuse, open-ended orchestration |
| **L3** | Multi-Script | Multi-script coordination and local orchestration | Inter-script interface agreements | Script collaboration within same task family | Organization-level reusable generic capabilities |
| **L4** | Skill | Productized encapsulation of scripts/templates/LLM calls | Manifest + schema + runtime + validator | High-frequency, reusable, semantically stable capability units | Long-cycle resource coordination, complex policy decisions |
| **L5** | Multi-Skill | Stable combination of multiple skills | Combination contract + dependencies + upgrade conditions | Groups of frequently co-occurring skill clusters | Independent role division, complex review coordination |
| **L6** | Agent | Stateful orchestration and decision responsibility | Lifecycle, state machine, upgrade interface | Tasks requiring retry, judgment, resource coordination | General-purpose entities without clear role boundaries |
| **L7** | Multi-Agent | Multi-role coordination and mutual review structure | Role boundaries, communication protocols, arbitration mechanisms | Parallel roles, context isolation, mutual review, organizational division | Scenarios that are merely "large tasks" without role differentiation |

These eight forms can be compressed into three categories:
- Levels 0-3: More focused on **implementation assets**
- Levels 4-5: More focused on **capability assets**  
- Levels 6-7: More focused on **organizational assets**

As systems move up the ladder, they increasingly do more than just "get things done"—they must simultaneously ensure:
- Tasks can be repeated
- Capabilities can be called by others
- Results can be validated
- Processes can be reviewed
- Operations can be observed
- Assets can be upgraded or split

This represents the fundamental difference between "assets" and "temporary implementations."

### 2.3 Key Claim: Level Differences Lie in Governance, Not Capability

Many debates mistakenly assume that differences between temporary scripts, skills, agents, and multi-agent systems primarily concern "who is more capable." EAPF advances a key claim:

> The essential difference between execution asset levels lies not in execution expressiveness, but in governance expressiveness.

In other words:
- Level 0 can produce a result but expresses minimal governance
- Level 4 can produce the same result but adds schema, reuse, validation, and callability
- Level 6 can produce the same result but adds state, responsibility, exception handling, and role boundaries
- Level 7 can produce the same result but adds role differentiation, context isolation, mutual review, and arbitration

This claim yields two important corollaries:

1. **One cannot only look at "whether it was produced" but must examine "at which level it was produced"**  
   Otherwise the system will continue rewarding lowest-level paths.

2. **High-level design cannot rely on unrestricted model improvisation but requires institutional system support**  
   Because governance expression is not a byproduct of natural generation but an explicitly designed structure.

All subsequent promotion rules in this paper therefore revolve around a single standard:

> Should the current task be elevated from an "implementation problem" to an "asset problem" or "organizational problem"?

## 3. The Execution Asset Promotion Framework (EAPF)

### 3.1 Four Core Principles

This entire methodology can be compressed into four core principles:

#### Principle One: High-Level First

When a task arrives at the system, default to matching the **highest available existing asset level**. If a task can already be completed by a skill, it should not revert to a temporary script. If a task requires agent coordination, a single skill should not bear the entire burden.

The significance of this principle: **The system must default to rewarding "reuse of existing high-level assets" rather than "repeating work at lower levels."**

#### Principle Two: Low-Level Restricted

Low-level assets are not prohibited but must be strictly limited to clear boundaries:
- Temporary scripts should only be used for one-time, low-risk, low-reuse-expectation tasks
- Execution models should not possess unrestricted low-level script-writing authority
- Lower-level assets should increasingly come with expiration, upgrade, or review requirements

The significance of this principle: **The goal is not to eliminate low-level assets but to prevent them from consuming higher-level ones.**

#### Principle Three: Failure Explicit

Any of the following conditions must be recorded as explicit events rather than silently accepted:
- Bypassing existing skills
- Repetition of temporary scripts
- Skill insufficiency
- Skill over-complication
- Review bypasses
- Multi-step tasks force-fit into single-level assets

Only when failures are explicit can the system truly evolve. Otherwise, all defects will be masked behind "tasks that appear completed."

#### Principle Four: Promotion by Design

Asset promotion cannot rely on inspiration but must depend on institutional triggers:
- Reuse frequency triggers
- Failure pattern triggers
- Review discovery triggers
- Dependency clustering triggers
- Organizational boundary triggers

**Conclusion**: Promotion is not an optional optimization but a system survival mechanism.

### 3.2 Diagnostic Expression

EAPF uses the following diagnostic expression to reason about system evolution direction:

```
System Evolution Rate = Asset Promotion Capture Rate × Reuse Density × Failure Explicitness × Review Intervention Rate - Temporary Script Escape Rate
```

Where:
- **Asset Promotion Capture Rate**: Whether the system can promptly absorb repetitive tasks into higher-level assets
- **Reuse Density**: Whether the same capability is stably reused across multiple tasks/projects
- **Failure Explicitness**: Whether bypasses, defects, boundary violations, and low quality are explicitly recorded
- **Review Intervention Rate**: Whether review, validator, and management can timely intervene at critical points
- **Temporary Script Escape Rate**: How many tasks that should be absorbed by high-level assets still escape to low levels

The engineering significance of this expression:
- If the first four items are weak, the system becomes increasingly fragmented
- If the last item is high, the system continuously degrades
- Only when the system simultaneously improves the first four items while reducing the last can it truly evolve from "script piles" to "skill systems, agent systems, multi-agent systems"

This is why **fine-tuning alone, harness design alone, or writing several skills alone** is insufficient: these interventions need to sit inside a unified evolution mechanism.

### 3.3 Five Closed Loops

The Execution Asset Promotion Framework is not a static structure but a dynamic system involving five interconnected closed loops:

1. **Execution Loop**: Tasks arrive, the system determines which asset level should handle them, executes, validates, and produces results.

2. **Exception Loop**: Any bypasses, failures, repeated low-level implementations, or review bypasses must enter an exception ledger rather than being silently accepted.

3. **Review Loop**: Review agents, human architects, or management agents check not only "whether the result is correct" but also "whether the path is correct." Path errors must trigger upgrade discussions.

4. **Promotion Loop**: Exception ledgers, reuse statistics, dependency clustering, and review discoveries all enter Promotion Review to decide whether to:
   - Upgrade asset levels
   - Split skills
   - Merge skill clusters
   - Introduce agents
   - Introduce multi-agent systems

5. **Training Loop**: All bypasses and failures must not remain at the governance level but must also feed back into:
   - Small model training data
   - Constraint layer rules
   - Skill manifests
   - Validators and review templates

Thus, EAPF is not "just an architecture diagram" but a **long-term evolution mechanism combining execution, review, promotion, and training**.

## 4. Asset Promotion Rules: When, How, and to What Level

### 4.1 From Temporary Script to LLM Script

When a task remains one-time but clearly involves fuzzy judgment, open-ended generation, or complex transformations, it should no longer stay in pure deterministic temporary script territory.

**Promotion Triggers**:
- Tasks require language understanding, semantic matching, interpretation, summarization, or generation
- Temporary scripts begin containing large hard-coded prompt strings
- Similar tasks need elastic changes across different inputs rather than fixed rules

**Post-Promotion Requirements**:
- Define input parameters and output formats explicitly
- Encapsulate LLM calls as separate script entry points
- Establish minimum constraints for prompts and output parsing

The goal of this step is not "to implement a big system" but to isolate "fuzzy capability" from temporary logic.

### 4.2 From LLM Script to Pipeline

When an LLM script begins to be stably accompanied by preprocessing, post-processing, validation, retry, or export actions, it is no longer a single-step capability and should be promoted to pipeline level.

**Promotion Triggers**:
- The same process stably contains 3+ fixed sequential steps
- Each execution requires identical preparation/cleanup/validation actions
- Humans frequently chain multiple scripts manually in a fixed order

**Post-Promotion Requirements**:
- Define step sequences
- Define intermediate states and failure points
- Define minimum checkpoints and retry logic

The focus of this step is: **to transform "repetitive sequences" from implicit operational habits into explicit process structures.**

### 4.3 From Pipeline to Multi-Script

When multiple pipelines or script fragments begin to repeat collaboration within the same task family, with emerging branching, reuse, and replacement requirements, promotion to multi-script level is warranted.

**Promotion Triggers**:
- A pipeline begins repeatedly calling other script modules
- Similar processes need switching between multiple variants
- Different projects begin reusing the same group of script clusters

**Post-Promotion Requirements**:
- Standardize inter-script interfaces
- Extract shared modules
- Define combination boundaries rather than allowing process files to expand infinitely

The essence of multi-script is moving from "single processes" toward "local script ecosystems."

### 4.4 From Multi-Script to Skill

When a script cluster already has stable semantic boundaries, cross-task reuse value, and clear input/output, it should no longer remain in engineer-private script libraries but be promoted to skill level.

**Promotion Triggers**:
- The same capability repeatedly appears across multiple tasks or projects
- Users or upper-level agents can already stably point to it with natural language
- Relatively stable input parameters, output artifacts, and success criteria already exist
- Reviews repeatedly mention "this should be encapsulated as a skill"

**Post-Promotion Requirements**:
- Add manifest
- Add `input_schema` / optional `output_schema`
- Add runtime entry point
- Add validator / review hook
- Add trigger conditions and upgrade conditions

The birth of a skill means: **A capability is formally acknowledged by the system as a "callable asset" rather than "a set of files the author knows how to use."**

### 4.5 From Skill to Multi-Skill

If multiple skills appear in fixed combinations over time, the issue is no longer just "existence of individual capabilities" but "existence of stably collaborative capability clusters."

**Promotion Triggers**:
- Multiple skills always appear together
- Stable dependencies exist between skills
- They collectively serve the same type of business outcome
- Calling individual skills places excessive coordination burden on upper layers

**Post-Promotion Requirements**:
- Define combination contracts for the skill cluster
- Define cluster-level input/output and failure upgrade conditions
- Clarify what remains handled by skill internals versus elevated to the combination layer

Multi-skill does not simply stack skills together but elevates **stable co-occurrence relationships** into formal structures.

### 4.6 From Multi-Skill to Agent

When the problem extends beyond "capability combination" to require sustained state, exception judgment, resource coordination, sequence selection, or upgrade decisions, agent introduction becomes appropriate.

**Promotion Triggers**:
- Long-term state maintenance is required
- Dynamic decisions on which skill to call based on context is required
- Exception handling, retry, rollback, or upgrade processing is required
- A relatively stable "role responsibility" must be assumed

**Post-Promotion Requirements**:
- Define clear role boundaries
- Define state machines
- Define callable asset scope
- Define upgrade and review interfaces

**Key Judgment**: An agent is not a "larger skill" but **an orchestrator assigned role responsibilities**.

### 4.7 From Agent to Multi-Agent

Multi-agent should only appear when problems genuinely rise to **organizational levels**, not merely because tasks are "large," contexts are "many," or situations feel "complex."

**Promotion Triggers**:
- Clear role specialization exists between agents
- Roles require relatively independent context windows
- Multiple subtasks can proceed in parallel
- Mutual review, arbitration, or cross-validation is required
- Resource coordination and global scheduling is required

**Post-Promotion Requirements**:
- Define role definitions
- Define communication protocols
- Define handoff artifacts and responsibility boundaries
- Define arbitration and conflict resolution mechanisms

Without these conditions, multi-agent often brings only additional overhead and harder-to-trace confusion.

### 4.8 Splitting, Merging, and Demotion Rules

Promotion is not one-way expansion. Healthy system evolution must also allow **splitting, merging, and demotion**.

**Cases requiring skill splitting**:
- A skill handles multiple weakly related tasks
- Input schema becomes increasingly disordered
- Reviews discover completely different failure modes
- Maintainers cannot articulate the skill's single responsibility

**Cases requiring multi-skill merging**:
- Multiple skills always appear together
- Individual skills are clear, but their combination relationship has stabilized
- Upper layers repeatedly orchestrate the same group of skills

**Cases requiring demotion**:
- An agent is actually just mechanically calling a skill
- A multi-agent system has no real role differentiation—just "multiple executioners"
- The governance costs of a complex level exceed its structural benefits

**Conclusion**: Truly healthy systems do not just escalate upward but can, at appropriate times:
- Upgrade
- Split
- Merge
- Demote

In other words, EAPF is not "making systems increasingly complex" but keeping systems at **the governance level most appropriate for current tasks**.

## 5. Skills: System Components, Not Documentation

### 5.1 The Four-Layer Skill Structure

In EAPF, a skill cannot merely equate to `SKILL.md`. A truly governable skill must contain at least four layers:

| Layer | Function | Necessity |
|-------|----------|-----------|
| **Manifest Layer** | Defines `name`, trigger conditions, `input_schema`, `output_schema`, upgrade conditions, callable runtime entry points | Hard constraint core |
| **Runtime Layer** | Scripts, templates, API wrappers, validators, review hooks, state management | True execution capability |
| **Playbook Layer** | Natural language descriptions, usage strategies, reference materials, experience rules | Helps models understand and trigger |
| **Telemetry Layer** | Logs, error codes, failure classification, review records, usage statistics | Supports subsequent upgrades and training |

If only Playbook exists without Manifest and Runtime, it resembles documentation rather than a skill.  
If only Runtime exists without Manifest and Telemetry, it resembles a script collection rather than a skill.  
Only when all four layers are complete does a skill truly become a system component.

### 5.2 Skill Strength Evaluation and Promotion Principles

Whether a skill is "strong" should not be judged by "occasional success" but by the following dimensions:
- Trigger boundaries are clear
- Input contracts are stable
- Outputs are verifiable
- Failures are attributable
- Runtime is complete
- Review is accessible

When a skill is insufficient, the correct approach is not to let execution agents bypass it, but to treat "skill insufficiency" as an explicit event and trigger **comprehensive analysis and upgrade**. Upgrade content typically includes:
- Completing the manifest
- Refactoring the runtime
- Adding validators
- Adding more explicit trigger descriptions
- Adjusting internal models, templates, or scripts called

**Key Principle**: Weak skills are not a reason for bypass but a signal for upgrade.

### 5.3 Skill Splitting Principles

A skill needs splitting typically not because it's "too large" but because it has lost clear single-capability boundaries. Judgment criteria include:
- Input parameters begin dividing into multiple weakly related clusters
- Output artifacts begin splitting into different types
- Failure modes are no longer consistent
- Users can no longer describe what it does in a single sentence

When splitting, prioritize cutting along these dimensions:
1. By input contract
2. By output artifacts
3. By failure modes
4. By responsibility attribution

The goal of splitting is not to make skills smaller but to allow each skill to regain clear, stable, verifiable capability boundaries.

### 5.4 Skill Clusters and Multi-Skill Formation Conditions

When multiple skills are already clear individually but their combination relationships become stable, skill clusters should form and consideration should be given to upgrading to multi-skill level.

Common signals include:
- These skills are always called together
- They serve the same type of result artifacts
- Sequential dependencies have stabilized
- The cost of repeatedly orchestrating them in upper layers is increasing

At this point, what needs elevation is not individual skills but **the relationships between these skills**.

## 6. Agents and Multi-Agents: Boundaries, Responsibilities, and Conditions

### 6.1 The Essence of Agents: Stateful Orchestration Roles, Not Larger Prompts

In this methodology, the essence of an agent is not "a smarter skill" but **a stateful role assigned orchestration responsibilities**.

An agent is called an agent not because it can engage in many conversational rounds, but because it assumes certain of the following responsibilities:
- Maintaining state
- Orchestrating multiple capability assets
- Judging whether to retry/upgrade/rollback
- Being responsible for result paths
- Explicitly handling exceptions

Therefore, something that only passively executes single skill calls should not be expansively called an agent—at most, it's part of skill runtime.

### 6.2 When Agents Should Be Introduced

When problems exhibit the following characteristics, single-agent introduction is appropriate:
- Dynamic orchestration across multiple skills is required
- Long-term state or task lifecycle maintenance is required
- Strategic exception handling is required
- Clear role responsibilities must be assumed
- Interaction with management or review layers is required

Typical examples include:
- Sequencer Agent
- Review Orchestration Agent
- Pipeline Recovery Agent
- Promotion Manager Agent

These are not simple capability blocks but "positions" with ongoing responsibilities.

### 6.3 When Multi-Agent Should Absolutely Not Be Used

The most common misuse of multi-agent is mistaking "large tasks" for "should use multi-agent."

The following scenarios typically should **not** use multi-agent:
- It's just that a skill hasn't been written well yet
- It's just that a single agent's context is too long
- It's just that tasks have many steps but no role differentiation
- It's just "opening multiple executioners to try their luck"
- It's just using complex structures to mask unclear skill boundaries

If the problem's essence is still "insufficient capability encapsulation" or "unclear single-role orchestration," address skills or single-agents first rather than immediately adopting multi-agent.

### 6.4 When Multi-Agent Must Be Used

Multi-agent is only necessary when problems genuinely rise to organizational levels.

Typical necessary conditions include:
- Clear role specialization exists
- Multiple roles need independent contexts
- Subtasks can proceed in parallel
- Mutual review, arbitration, or cross-validation is required
- Global resource coordination is required
- Different permissions or risk domains need isolation

Multi-agent can be understood as:

> When a single role cannot complete tasks within the same context and responsibility boundaries, the system needs to enter the organizational collaboration stage.

This is why multi-agent should not be considered a "default advanced mode" but **the final organizational structure layer**.

## 7. Harness and Permission Design: Forcing Systems to Evolve Along High-Level Paths

### 7.1 Separation of Policy and Execution Models

One of EAPF's most critical engineering requirements is separating "what to think" from "what to do."

Specifically, the execution layer should be divided into two roles:
- **Policy Model**: Only responsible for determining which asset level should handle the current task, and whether clarification, escalation, or rejection is needed
- **Execution Runtime**: Actually executes skills, pipelines, scripts, validators, and review processes

The policy model should not be designed as a "versatile worker." Its ideal output is not code but restricted protocols, such as:

```json
{
  "decision": "call_skill | clarify | escalate | reject",
  "target": "...",
  "reason_code": "..."
}
```

Once a policy model can still directly write scripts, modify files, or arbitrarily construct commands, it retains the shortcut to the lowest level, and the entire asset promotion framework becomes short-circuited.

### 7.2 Capability-Based Permissions

This paper recommends a **capability-based permission model**:
- Execution Protocol Model: No direct shell write permissions, no direct file write permissions, can only output restricted protocols
- Skill Runtime: Only possesses minimum permissions required to complete the skill
- Review Agent: Primarily has read, review, and comment permissions, not default execution write permissions
- Management/Promotion Layer: Has upgrade, arbitration, and process adjustment permissions but does not directly replace underlying execution

The core of permissions is not "who is more capable" but "who should assume which level of responsibility."  
Only when permissions are sliced according to responsibility boundaries can the system truly elevate the default execution level.

### 7.3 Minimum Essential Harness Components

To force systems to evolve along high-level paths, a qualified harness must include at least these components:

| Component | Responsibility |
|-----------|----------------|
| **Asset Registry** | Records all script/pipeline/skill/agent/multi-agent assets and their contracts |
| **Policy Layer** | Determines which asset level should be called based on tasks |
| **Runtime Executor** | Executes approved runtime entry points |
| **Validator Layer** | Checks input/output, result quality, and constraint satisfaction |
| **Review Layer** | Reviews "whether results are correct" and "whether paths are correct" |
| **Telemetry Ledger** | Records usage, failures, bypasses, upgrades, and splits |
| **Promotion Engine** | Proposes or executes asset upgrades/splits/merges based on logs and rules |

If the latter three components are missing, the system can only "execute" but not "evolve."

### 7.4 Violation Capture and Forced Escalation

In EAPF, the following behaviors should be treated as violation events rather than "flexible handling":
- Writing temporary scripts directly when existing skills are available
- Not escalating when skills fail, but bypassing directly
- Forcing multi-step tasks into single layers when multiple skills are clearly needed
- Skipping review, validator, or established validation processes
- Disguising low-level temporary implementations as high-level asset successes

These events must enter a unified exception ledger with reason codes such as:
- `BYPASS_EXISTING_SKILL`
- `MISSING_SKILL_RUNTIME`
- `SKILL_TOO_BROAD`
- `SEQUENCE_REQUIRED`
- `SKIP_REVIEW_REQUESTED`

**Principle**: As long as the system allows silent success of violations, high-level assets cannot truly grow.

## 8. Training and Alignment: What Small Models Should Be Trained To

### 8.1 Small Models as Execution Protocol Models, Not Code Workers

The most suitable role for small models is not "low-capability version of large model programmers" but **execution protocol models**.

Their most important responsibility is not writing good code but:
- Identifying which asset level the current task should follow
- Judging whether parameters are insufficient
- Judging whether escalation to higher levels is mandatory
- Judging whether system-external tasks should be rejected
- Maintaining compliance with existing asset paths

This means small model training targets should shift from "code capability" to "protocol capability."

### 8.2 Correct Training Objectives

Correct training objectives are not having models output arbitrary natural language explanations but having them stably output restricted execution protocols.

Ideal training actions include:
- `call_skill`
- `clarify`
- `escalate`
- `reject`

Wrong training directions include:
- Letting small models write business code themselves
- Letting small models decide multi-step orchestration details themselves
- Letting small models freely improvise implementation when skills are insufficient

Once training objectives deviate, models will be trained as "low-cost temporary script generators" rather than system components.

### 8.3 Data Sources: From Real Failures

The most valuable training data does not come from manually imagined perfect examples but from real system failures and bypass events.

High-value data sources include:
- Bypass violation logs for skills
- Path errors discovered by reviews
- Upgrade records from skill insufficiency
- Clarification cases from missing parameters
- Cases where multi-step tasks were incorrectly compressed into single steps
- Cases of multi-agent misuse

In other words, training data should directly come from:

> Where the system actually attempts to degrade toward lower levels.

### 8.4 Evaluation Metrics: Not "Whether They Can Write" but "Whether They Follow Rules"

This paper suggests core evaluation metrics centered on governance quality rather than text quality:

| Metric | Meaning |
|--------|---------|
| **Protocol Compliance Rate** | Whether output falls within allowed decision space |
| **Escalation Recall Rate** | Whether models truly escalate when they should |
| **Missing Parameter Clarification Rate** | Whether models choose clarification over speculation when parameters are insufficient |
| **Bypass Failure Rate** | Whether models resist temptation when induced to "just write scripts" |
| **Asset Matching Accuracy** | Whether models select the correct asset when skills should be called |
| **Temporary Script Recurrence Rate** | Whether similar tasks continuously escape to the lowest level |

Such metrics better reflect whether models have truly become part of the system rather than free executors that can escape institutional control whenever a shortcut appears.

## 9. Organizational Governance: Continuous System Evolution Rather Than Degradation

### 9.1 Three Ledgers

To make EAPF function, the system must maintain at least three ledgers:

#### Ledger One: Asset Registry

Records all:
- Temporary scripts
- LLM-calling scripts  
- Pipelines
- Multi-scripts
- Skills
- Multi-skills
- Agents
- Multi-agents

With their ownership, versions, contracts, success rates, call frequencies, and upgrade status.

#### Ledger Two: Exception Ledger

Records all:
- Bypass events
- Skill failure events
- Review interception events
- Missing parameter events
- Multi-step task compression events
- Permission boundary violation events

The value of the exception ledger is making the system understand **where it continuously degrades**.

#### Ledger Three: Promotion Backlog

Records all pending upgrade, split, merge, and downgrade candidates with:
- Trigger reasons
- Target level candidates
- Responsible persons
- Risks
- Expected benefits

Without these three ledgers, the system has only execution, no evolution.

### 9.2 Review Agent and Management Agent Responsibilities

In this methodology, review and management are not decorations but key nodes in the upgrade loop.

**Review Agent Responsibilities**:
- Check result quality
- Check whether execution paths comply
- Discover unclear skill boundaries or insufficient capabilities
- Route issues to exception and promotion ledgers

**Management Agent Responsibilities**:
- Decide whether to upgrade, split, merge, or downgrade
- Decide whether agents or multi-agent systems need introduction
- Decide which issues should enter training feedback

In other words:
- Execution layer is responsible for getting things done
- Review layer is responsible for exposing problems
- Management layer is responsible for upgrading the system

### 9.3 Key Metric System

This paper suggests tracking at least these metrics:

| Metric | Meaning |
|--------|---------|
| **Temporary Script Escape Rate** | Proportion of tasks that should be handled by high-level assets but escape to L0 |
| **Asset Promotion Capture Rate** | Proportion of repetitive tasks formally upgraded |
| **Skill Reuse Density** | How many tasks/projects a single skill stably reuses |
| **Bypass Interception Rate** | Review/harness capability to discover violation paths |
| **Upgrade Latency** | Time from problem occurrence to asset upgrade completion |
| **Skill Split Debt** | Number of visibly over-broad skills not yet split |
| **Multi-Agent Misuse Rate** | Proportion of cases where multi-agent was inappropriately used |

These metrics together describe not "how smart models are" but "whether systems can self-govern."

### 9.4 Periodic Promotion Review

Promotion Review should be treated as a regular mechanism, not occasional meetings. It should at least periodically answer four questions:
1. Which temporary implementations actually should be upgraded?
2. Which skills actually should be split?
3. Which skill clusters already should be upgraded to agents?
4. Which agent structures are overestimated and should be merged or downgraded?

**Key Principle**: Without periodic Promotion Review, systems can only passively accumulate rather than actively evolve.

## 10. Relationship to DACES/HIL-5/MCP and AES

The Execution Asset Promotion Framework builds upon and extends several foundational frameworks in multi-agent systems:

### 10.1 Relationship to DACES/HIL-5/MCP

EAPF operates as a governance layer around DACES, HIL-5, and MCP-style coordination. It does not replace these frameworks; it defines how execution assets should be selected, upgraded, constrained, and reviewed so that coordination mechanisms remain structurally healthy:

- **DACES**: EAPF provides the asset promotion logic that prevents individual agents from bypassing the decentralized architecture
- **HIL-5**: EAPF institutionalizes human oversight through explicit review loops and promotion mechanisms
- **MCP**: EAPF ensures that model context usage follows proper asset boundaries rather than arbitrary low-level scripting

### 10.2 Relationship to AES (Agent-Executable Specification)

EAPF represents a theoretical layer that can extend AES from an execution specification into a governance-aware asset lifecycle:

- **Beyond mere execution**: AES describes task contracts and execution reports, while EAPF asks which asset level should handle the task
- **Beyond individual nodes**: AES can carry task state and constraints, while EAPF manages the asset ecosystem around those tasks
- **Beyond static capability use**: AES supports execution, while EAPF defines the review and promotion loops that improve future execution

### 10.3 Integration Strategy

The most effective multi-agent systems integrate these frameworks hierarchically:

```
EAPF (Governance Layer)
    ↓
DACES/HIL-5/MCP (Coordination Layer)
    ↓  
AES (Execution Layer)
```

This integration ensures that coordination frameworks function within proper governance constraints and execution systems evolve according to institutionalized promotion rules.

## 11. Limitations and Open Questions

### 11.1 Current Limitations

The EAPF framework has several current limitations:

1. **Complexity Overhead**: The full eight-form model may impose excessive complexity for simple use cases or small-scale projects
2. **Promotion Threshold Definitions**: While the framework identifies promotion triggers, precise quantitative thresholds remain implementation-specific
3. **Multi-Agent Coordination**: The framework provides less detailed guidance on the specific protocols and arbitration mechanisms for complex multi-agent scenarios
4. **Dynamic Environments**: Current implementation focuses on relatively stable asset definitions; adaptation to highly dynamic environments requires further research

### 11.2 Open Research Questions

Several important questions remain open for future investigation:

1. **Automatic Promotion Detection**: Can machine learning techniques automatically detect promotion opportunities from system telemetry without requiring explicit rule definitions?
2. **Cross-System Asset Migration**: How should assets be promoted and transferred across different multi-agent systems with potentially conflicting conventions?
3. **Human-AI Collaboration in Promotion**: What are the optimal interfaces for human operators to collaborate with AI systems in promotion decisions?
4. **Failure Recovery Strategies**: Beyond explicit failure recording, what systematic strategies can be developed for automatic recovery and reintegration of degraded assets?
5. **Scalability Limits**: What are the practical scalability limits of the eight-form model, and how might it adapt to extremely large-scale multi-agent systems?

### 11.3 Future Evolution Directions

The EAPF framework is intended to evolve through both theoretical refinement and practical implementation experience:

1. **Domain-Specific Adaptations**: Development of industry-specific variations tailored to software engineering, scientific research, business processes, and other domains
2. **Temporal Asset Management**: Integration of time-based considerations for asset lifecycle management and obsolescence planning
3. **Cross-organizational Standards**: Potential development of industry standards for asset manifests and promotion protocols
4. **Automated Compliance Checking**: Development of tools that automatically verify compliance with EAPF principles during system operation

## 12. Conclusion

### 12.1 Core Insight

The fundamental insight of the Execution Asset Promotion Framework is:

> Multi-agent system construction should not focus on making agents increasingly resemble versatile engineers, but on making systems increasingly resemble organizations that continuously upgrade execution assets.

The most important differences between temporary scripts, pipelines, skills, agents, and multi-agent systems do not lie in "who is more capable" but in "who assumes higher-level governance responsibilities."

### 12.2 Methodological Summary

If this entire methodology is compressed into a single phrase, it is:

> **High-level first, low-level restricted, failure explicit, promotion institutionalized.**

If expanded into a system closed loop, it becomes:

> Task arrival → Match highest available asset → Execute and validate → Record violations and exceptions → Trigger Promotion Review → Upgrade/split/merge/demote → Feed back training and constraints → Re-enter next execution cycle.

### 12.3 Fundamental Implications for Multi-Agent Systems

For multi-agent systems, the most fundamental implication is not "whether to use multi-agent" but:

1. First govern low-level assets well before rushing to high-level structures
2. First make skills hard components before discussing agent collaboration
3. First make failures explicit before discussing system optimization
4. First make promotion institutional before discussing self-evolution

Therefore, this paper does not offer a local technique but a comprehensive methodology:

> Do not pin system hopes on agent self-awareness, but build systems on asset maturity ladders, permission boundaries, review mechanisms, and upgrade loops.

This is the fundamental engineering path from "temporary script piles" to "skill systems, agent systems, multi-agent systems."
