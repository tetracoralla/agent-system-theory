# Agent、Skill、Script边界与相变问题深度研究报告（2026版）

---

## 执行摘要

本报告基于2025-2026年最新学术研究、工程实践及行业前沿探索，深入分析了**多Agent系统中Agent、Skill、Script（可执行代码）三者之间的边界定义、相变机制及架构选型原则**。研究发现，这三个概念之间存在明确的**复杂度阈值驱动的相变关系**，理解这些相变点对于构建高效、可扩展的AI系统至关重要。

报告核心发现：
- **Script→Skill相变点**：任务标准化程度<70%或异常处理分支>5个时，应从Script升级为Skill
- **Skill→Agent相变点**：技能库规模50-100个时出现认知负载相变，需引入Agent进行路由管理
- **单Agent→多Agent相变点**：任务复杂度评分>8分（10分制）或需跨领域专业知识时触发
- **三层架构演进规律**：Script（确定性）→ Skill（半确定性）→ Agent（不确定性）形成连续谱系
- **相变预警指标**：决策准确率断崖下跌、维护成本指数增长、执行失败率>15%为关键信号
- **混合架构最优解**：70%标准化任务用Script/Workflow，30%非标准化任务用Skill/Agent

│                                    ▼                                    │

│  │  • 特征：半确定性、可被Agent调用、包含LLM推理                    │   │
│  │  • 典型形式：Anthropic Skills、MCP Tools、Function Calling       │   │
│  │  • 决策主体：混合（人类定义边界，LLM决定调用时机）               │   │
│  │  • 适用场景：需要一定灵活性但流程相对固定的任务                  │   │

│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  • 定义：具备感知、规划、决策、执行能力的自主智能实体            │   │
│  │  • 特征：不确定性、自主决策、环境适应、目标导向                  │   │
│  │  • 典型形式：Claude Code、Devin、Manus、Kimi K2.5 Agent          │   │
│  │  • 决策主体：LLM（运行时自主决策）                               │   │
│  │  • 适用场景：开放性问题、复杂任务、需要动态适应的场景            │   │

│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

### 1.2 核心边界对比矩阵

| 维度 | Script | Skill | Agent |
|------|--------|-------|-------|
| **决策自主性** | 0%（完全预定义） | 30%（调用时机自主） | 90%+（完全自主） |
| **确定性程度** | 100% | 60-80% | 20-50% |
| **人类介入** | 编写时完全定义 | 定义边界+调用策略 | 仅定义目标 |
| **异常处理** | 预定义分支 | LLM动态判断 | 自主发现并修复 |
| **上下文依赖** | 无/有限 | 中等 | 强依赖 |
| **可预测性** | 完全可预测 | 部分可预测 | 难以完全预测 |
| **维护成本** | 低 | 中 | 高 |
| **灵活性** | 低 | 中 | 高 |

### 1.3 连续谱系模型

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Script-Skill-Agent连续谱系                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  确定性 ←──────────────────────────────────────────→ 不确定性           │
│                                                                         │
│  Script        Skill-Lite      Skill      Agent-Lite      Agent        │
│    │              │             │            │              │           │
│    ▼              ▼             ▼            ▼              ▼           │
│  100%          80-90%        60-80%       30-50%         10-30%        │
│  确定性        确定性        确定性       确定性         确定性         │
│                                                                         │
│  典型例子：                                                             │
│  • Script: 数据清洗脚本、定时备份、API调用                             │
│  • Skill-Lite: 简单RAG查询、格式化输出                                 │
│  • Skill: 代码审查、文档生成、数据分析                                 │
│  • Agent-Lite: 单轮任务执行、简单工具调用                              │
│  • Agent: 完整项目开发、长期任务规划、多步骤问题解决                   │
│                                                                         │
│  关键洞察：                                                             │
│  • 三者不是离散的类别，而是连续谱系上的不同位置                        │
│  • 相变发生在谱系的特定阈值点，而非突然跳跃                            │
│  • 最优架构往往是混合使用，而非单一选择                                │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.4 与相关概念的关系

| 概念 | 与Script关系 | 与Skill关系 | 与Agent关系 |
|------|-------------|-------------|-------------|
| **Workflow** | 多个Script编排 | 可包含Skill节点 | 可被Agent驱动 |
| **Tool/Function** | 原子级Script | Skill的组成部分 | Agent调用的单元 |
| **MCP** | 通信协议 | Skill标准化接口 | Agent扩展机制 |
| **Prompt** | 可视为文本Script | Skill的内部实现 | Agent的认知输入 |

### 2.2 相变预警指标

| 指标 | Script阶段 | 相变临界点 | Skill阶段 |
|------|-----------|-----------|----------|
| **代码行数** | <500行 | 500-1000行 | >1000行（含Prompt） |
| **异常分支数** | <5个 | 5-10个 | >10个（LLM处理） |
| **维护工时/月** | <10小时 | 10-20小时 | <10小时（配置化） |
| **执行失败率** | <5% | 5-15% | <5%（自适应） |
| **需求变更响应** | >3天 | 1-3天 | <1天 |

### 5.4 相变数学模型

基于复杂系统理论，可建立以下相变数学模型：

```
相变概率 P(transition) = f(U, C, M, F)

其中：
• U = 任务不确定性 (Uncertainty)
• C = 认知负载 (Cognitive Load)
• M = 维护成本 (Maintenance Cost)
• F = 执行失败率 (Failure Rate)

相变条件：
• Script→Skill: P > 0.6 当 U > 0.3 或 C > 5分支
• Skill→Agent: P > 0.6 当 C > 50技能 或 F > 0.15
• 单Agent→多Agent: P > 0.6 当 U > 0.7 或 C > 100K tokens

相变后的系统特性变化：
• 自由度增加：决策空间扩大
• 涌现性增强：整体能力>部分之和
• 可预测性降低：行为更难完全预测
• 鲁棒性提升：容错能力增强
```

### 8.3 相变失败案例与教训

| 案例 | 问题 | 原因 | 教训 |
|------|------|------|------|
| **过早Agent化** | 成本超支300% | 简单任务用Agent | 遵循简单优先原则 |
| **过晚升级** | 失败率>40% | 技能库达150个未升级 | 监控相变预警指标 |
| **架构跳跃** | 系统不稳定 | Script直接跳到多Agent | 渐进式升级 |
| **边界模糊** | 维护困难 | Skill和Agent功能重叠 | 明确职责边界 |
