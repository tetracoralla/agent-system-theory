# 执行资产升级框架中文导读：从临时脚本到多 Agent 系统

本文是英文论文 [Execution Asset Promotion Framework: From Scripts to Multi-Agent Systems](./3.9.14_Execution-Asset-Promotion-Framework.md) 的中文导读，而不是逐字翻译。英文论文是正式引用版本；本文用于帮助中文读者快速理解核心概念、术语和问题意识。

## 一句话总结

**多 Agent 系统的关键，不是让 Agent 越来越像万能工程师，而是让系统能够持续把临时执行沉淀为可治理、可复用、可审查、可升级的执行资产。**

这就是 Execution Asset Promotion Framework，简称 EAPF。

## 为什么需要 EAPF？

很多 Agent 系统在设计图上看起来很高级：有 Planner，有 Executor，有 Reviewer，有 Tool，有 Skill，有 Workflow，甚至有 Multi-Agent。

但一旦进入真实执行，常见情况是：

- Agent 不调用已有 Skill，而是临时写脚本；
- 重复任务没有沉淀成 Pipeline；
- Pipeline 没有升级成可复用 Skill；
- Skill 失败后，Agent 绕过 Skill 自己实现；
- 明明需要角色分工，却让单个 Agent 硬扛；
- 明明只是 Skill 边界不清，却误用 Multi-Agent 掩盖问题。

这类现象在 EAPF 中被称为：

> **资产层级堕落（Asset-Level Collapse）**

它的危险之处在于：短期看，任务可能完成了；长期看，系统没有成长，只是在不断堆积临时路径。

## 核心命题

EAPF 的核心判断是：

> `临时脚本 → LLM 调用脚本 → Pipeline → Multi-Script → Skill → Multi-Skill → Agent → Multi-Agent`  
> 不是一组并列工具，而是一条执行资产成熟度阶梯。

越往上，不只是“能力更强”，而是治理语义更多：

- 输入输出更明确；
- 复用边界更清楚；
- 验证机制更稳定；
- 失败更可追踪；
- 权限更可控制；
- 责任更可归属；
- 系统更容易演化。

因此，EAPF 关心的不是“这件事有没有做出来”，而是：

> **它是沿着正确的执行资产路径做出来的吗？**

## 八类执行资产

| 层级 | 名称 | 核心含义 |
| --- | --- | --- |
| L0 | Temporary Script | 一次性、低风险、无复用预期的临时脚本 |
| L1 | LLM Script | 封装单步模糊处理能力的 LLM 调用脚本 |
| L2 | Pipeline Script | 多个稳定步骤组成的固定流程 |
| L3 | Multi-Script | 多脚本协作形成的局部脚本生态 |
| L4 | Skill | 有 manifest、schema、runtime、validator 的可调用能力组件 |
| L5 | Multi-Skill | 多个 Skill 的稳定组合 |
| L6 | Agent | 有状态、职责、判断和升级接口的岗位型执行者 |
| L7 | Multi-Agent | 多角色协同、互审、仲裁和上下文隔离的组织结构 |

这里最重要的一点是：

> Multi-Agent 不是“任务变大了就上”，而是当问题真正进入组织分工、角色隔离、互审仲裁时才需要上。

## 四条原则

EAPF 可以压缩成四句话：

1. **High-Level First：高层优先**  
   如果已有更高层资产可以处理任务，系统应优先调用它，而不是退回临时脚本。

2. **Low-Level Restricted：低层受限**  
   临时脚本不是禁止使用，但必须限定在一次性、低风险、低复用预期的场景。

3. **Failure Explicit：失败显性**  
   绕过 Skill、重复写脚本、跳过 Review、强行压缩多步任务，都应该被记录为治理事件。

4. **Promotion Institutionalized：升级制度化**  
   资产升级不能靠 Agent 灵光一现，而要靠复用频率、失败模式、审查发现和治理流程触发。

## 为什么 Agent 不会自然升级资产？

因为“升级”不是普通推理行为，而是治理行为。

从临时脚本升级到 Skill，不只是把代码写得更漂亮，而是要补齐：

- manifest；
- input/output schema；
- runtime；
- validator；
- telemetry；
- review hook；
- 所有权和维护边界。

这些收益通常是长期、全局、系统级的；而 Agent 在单次任务中更容易选择局部最短路径。

所以，EAPF 不把希望寄托在 Agent 自觉上，而是要求系统设计出正式的升级路径。

## Skill 在 EAPF 中是什么？

EAPF 反对把 Skill 理解成一段 `SKILL.md` 提示词。

一个真正的 Skill 至少应包含四层：

| 层 | 作用 |
| --- | --- |
| Manifest Layer | 定义名称、触发条件、输入输出 schema、运行入口和升级条件 |
| Runtime Layer | 脚本、模板、API 包装、validator、状态管理 |
| Playbook Layer | 给人和模型看的使用说明、经验规则、参考材料 |
| Telemetry Layer | 日志、错误码、失败分类、使用统计和审查记录 |

只有 Playbook 没有 Runtime，它更像文档。  
只有 Runtime 没有 Manifest 和 Telemetry，它更像脚本集合。  
四层齐备，Skill 才真正成为系统组件。

## 对小模型训练的启示

EAPF 也改变了小模型在 Agent 系统中的定位。

小模型不应该被训练成“便宜版程序员”，而应该被训练成：

> **执行协议模型（Execution Protocol Model）**

它的任务不是自由写代码，而是稳定输出受限决策：

- `call_skill`
- `clarify`
- `escalate`
- `reject`

评估小模型时，也不应只看“会不会写”，而应看：

- 是否遵守协议；
- 是否在参数不足时澄清；
- 是否在 Skill 不足时上报；
- 是否抵抗“直接写脚本”的诱导；
- 是否选择了正确资产路径。

## 和 DACES/HIL-5/MCP/AES 的关系

EAPF 不是替代 DACES、HIL-5、MCP 或 AES，而是给它们增加一个治理视角。

可以粗略理解为：

```text
EAPF：治理层，决定任务应该由哪一级执行资产处理，以及如何升级资产
DACES/HIL-5/MCP：协调层，决定状态、控制、人类介入和上下文协作方式
AES：执行规格层，承载任务契约、输入输出、约束和执行报告
```

如果说 AES 关心“任务如何被执行”，那么 EAPF 进一步追问：

> **这个任务本来应该由哪一级资产执行？执行路径本身是否合规？这次执行是否应该触发资产升级？**

## 适合谁读？

这篇论文适合：

- 正在构建多 Agent 系统的人；
- 正在设计 Skill / Tool / Workflow / Agent 边界的人；
- 想让 Agent 系统长期演化，而不是堆积临时脚本的人；
- 关心 Harness Engineering、AI 工作流治理、执行可靠性的人；
- 希望让其他 Agent 也能稳定理解系统边界的人。

## 不要误读为

EAPF 不是说：

- 临时脚本不能用；
- Skill 永远比脚本好；
- Agent 永远比 Skill 高级；
- Multi-Agent 是最终目标；
- 所有系统都要完整实现八层模型。

EAPF 真正强调的是：

> **系统应该始终处在与当前任务最匹配的治理层级。**

有时应该升级，有时应该拆分，有时应该合并，有时甚至应该降级。
