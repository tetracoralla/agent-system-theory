## 执行摘要

本文提出一套基于动态适应与协同演化理论的多Agent系统工程应用框架，旨在解决企业工作流智能化转型中的核心挑战：**如何系统化地实现从"一句话需求"到多Agent自主工作流执行的平稳过渡**。

框架以DACES理论为元框架，以Harness Engineering为工程方法论，以MCP/A2A协议体系为通信基础设施，以AES为任务级协作载体，以分层标准化为治理架构，以受控演化为安全保障，构建了从理论到实践的完整路径。系统的核心交互范式是：**用户仅需提出一句自然语言需求，系统即自主完成需求澄清、调研分析、多Agent工作流编排与执行，并在关键节点设置硬性卡点供用户评审，最终交付符合需求目标的成果**。

**v3.0修订核心增强**：

1. **AES任务级协作载体**：新增AES作为任务状态、背景、约束、交接、执行报告与断点恢复的统一载体，补足v2中"工作流节点承载物缺位"的核心问题
2. **A2A/MCP/AES三层分工**：明确A2A负责运行时消息协作与任务委派，MCP负责工具与外部能力连接，AES负责任务状态与断点恢复
3. **证据边界规范化**：清理所有未经充分核验的生态规模数据，区分"官方公开可核验""行业观察""经验假设"三级证据
4. **工作流节点重定义**：将节点从纯步骤模型改为 Task + AES + Actor + Checkpoint + Artifact 五元组，支持可交接、可恢复、可审计
5. **交付物体系升级**：卡点主工件升级为AES（.aes.yaml），Markdown作为人类可读派生件
6. **多Agent共建治理**：补充所有权、冲突策略、审批回执、工件哈希等企业级协作治理字段

**v4.0前沿增强**：

7. **Durable Execution 运行时语义**：引入 exit/async/sync 三级持久化模式、确定性重放、副作用隔离与补偿/Saga模式，为工作流执行提供平台级可靠性保障

9. **Eval-Driven Control Plane**：将评估从附属指标升级为控制闭环的一等公民，支持离线/在线/影子评估和门禁驱动决策

## 第一章 研究背景与问题陈述

### 1.1 企业工作流智能化转型的挑战

随着AI Agent技术的快速发展，企业面临从传统工作流向智能化工作流转型的历史机遇。MCP协议已捐赠Linux Foundation (AAIF)，A2A协议已获150+组织支持并转入Linux Foundation项目化。然而，现有实践暴露以下核心问题：

| 问题类别 | 具体表现 | 理论根源 |
|---------|---------|---------|
| **架构模糊性** | Agent边界不清，职责重叠，技能调用混乱 | 缺乏标准化架构定义 |
| **演化失控风险** | 系统自主演化可能导致安全边界突破 | 缺乏受控演化机制 |
| **适配性不足** | 通用框架难以适应企业特定工作流 | 缺乏分层配置机制 |
| **治理缺失** | 权限越权、问题隐瞒、责任不清 | 缺乏系统化治理框架 |
| **过渡困难** | 从人工主导到自主执行的跨越缺乏方法论 | 缺乏HIL分级过渡机制 |
| **交付质量失控** | Agent自主执行过程中缺乏有效的人审环节 | 缺乏硬性卡点机制 |
| **上下文污染** | 多Agent协作中信息无序共享导致性能下降 | 缺乏上下文隔离策略 |
| **可观测性缺失** | 分布式Agent执行难以追踪和诊断 | 缺乏可观测性架构 |
| **韧性能力不足** | 系统面对异常扰动恢复能力弱 | 缺乏韧性工程整合 |
| **任务连续性断裂** | 长任务中断后无法从断点恢复，跨Agent交接上下文丢失 | 缺少任务级状态载体（AES） |
| **工作流节点承载物缺位** | 节点仅定义步骤和依赖，缺少输入输出契约和恢复指令 | 缺少可交接可恢复的节点模型 |

### 1.2 "一句话需求"范式的提出

#### 1.2.1 概念定义

**"一句话需求"（One-Sentence Requirement）** 指的是用户以一句自然语言表达其需求意图，系统承担从需求澄清到最终交付的全部中间工作。这一范式的核心假设是：

> 用户只负责"想要什么"和"验收结果"，系统负责"怎么做"和"过程质量"。

这本质上是一个**HIL L4委托级**的交互模式——用户在关键决策节点被邀请参与评审，但过程的绝大部分由系统自主推进。

#### 1.2.2 与现有模式的对比

| 维度 | 传统开发模式 | 低代码平台 | AI Copilot | **一句话需求模式** |
|------|------------|-----------|-----------|------------------|
| 需求表达 | 需求文档PRD | 表单配置 | 对话式指导 | 一句话自然语言 |
| 需求分析 | 人工分析 | 模板匹配 | 辅助分析 | **自主完成** |
| 方案设计 | 人工设计 | 预设模板 | 辅助设计 | **自主设计+卡点评审** |
| 实现过程 | 人工编码 | 自动生成 | 辅助编码 | **自主编排执行（Task/Subagent）** |
| 质量控制 | 人工测试/QA | 内置校验 | 辅助测试 | **硬性卡点+自主验证** |
| 交付物 | 软件/系统 | 应用 | 代码片段 | **文档/软件/应用均可** |
| 人类参与 | 全程 | 配置时 | 持续对话 | **仅在卡点节点** |
| 断点恢复 | 人工接手 | 不可恢复 | 部分恢复 | **AES驱动的自动恢复** |
| 跨Agent交接 | 不适用 | 不适用 | 不适用 | **AES作为交接载体** |

#### 1.2.3 技术可行性基础

"一句话需求"范式的技术可行性建立在以下前沿成果之上：

1. **Harness Engineering**（OpenAI, 2026）：3名工程师5个月构建100万行代码（零手写），效率10倍提升
2. **Agent Lifecycle FSM**：动态角色管理节省40-60%计算资源，成为2026年行业标准
3. **选择性上下文共享**：选择性共享基准135%，远超强制隔离（100%）和完全共享（85%）
4. **Task/Subagent架构**：会话隔离机制可节省70-90%主会话上下文（文献3.2.6）
5. **A2A协议生态**：150+组织支持，提供标准化的Agent间通信基础设施（来源：Google Developers Blog、Linux Foundation公告）
6. **MCP协议生态**：数千个服务器实现（来源：Anthropic官方），已捐赠Linux Foundation (AAIF)
7. **AES任务级协作载体**：三层结构支持断点恢复、跨Agent交接和状态连续性（来源：AES README）

### 1.3 现有理论框架的局限

基于对现有理论框架的分析，识别以下关键理论空白：

1. **理论到实践的鸿沟**：DACES、HWAT等理论框架缺乏面向"一句话需求"场景的工程实现细节
2. **标准化与灵活性的矛盾**：过度标准化导致僵化，过度灵活导致混乱
3. **自主演化与系统稳定的冲突**：如何确保演化不破坏系统核心安全边界
4. **企业特异性处理缺失**：如何平衡通用框架与企业定制需求
5. **交付质量保障缺失**：多Agent自主执行过程中，如何在最小化人类干预的同时保障交付质量
6. **上下文管理策略不足**：多Agent协作中的上下文隔离与共享缺乏系统性方案
7. **动态边界管理缺失**：Agent边界应随任务需求动态调整
8. **韧性工程整合缺失**：缺乏与韧性工程四大能力的系统性映射
9. **可观测性架构缺失**：分布式Agent执行的可追踪性和可诊断性不足
10. **PDA测量体系缺失**：部署后自治能力的持续测量与HIL动态调整机制
11. **任务级协作载体缺失**：缺少统一的任务状态、交接和断点恢复承载物（v3新增识别）
12. **工作流节点可恢复性缺失**：节点不支持从任意中断点恢复或跨Agent接手（v3新增识别）

### 1.4 研究目标与贡献

**研究目标**：构建一套面向"一句话需求"范式的多Agent系统工程应用框架，实现从用户意图到最终成果的端到端自主执行，通过硬性卡点机制保障交付质量，通过受控演化机制确保系统安全，通过韧性工程增强系统可靠性，通过PDA测量体系支持持续优化，通过AES任务级协作载体保障任务连续性和断点恢复能力。

### 2.2 Harness Engineering：从约束到生产力

#### 2.2.1 Harness的核心定义

基于文献1.2、1.3、2.7.1的综合分析，Harness Engineering是OpenAI于2026年正式提出的工程方法论，核心理念是 **"Humans steer, agents execute"**（人类掌舵，Agent执行）。

**Harness在本框架中的角色定义**：
> Harness是包裹在Agent执行环境外的"马具层"，将不确定的模型行为转化为稳定、可预测、可审计的生产力输出。在"一句话需求"范式中，Harness承担三重职责：
> 1. **意图护栏**：确保Agent对用户需求的理解不偏离原始意图
> 2. **执行护栏**：确保Agent的每一步执行都在安全边界内
> 3. **质量护栏**：确保最终交付物符合质量标准

### 2.3 硬性卡点(Hard Checkpoint)机制

#### 2.3.1 卡点理论依据

硬性卡点机制是"一句话需求"范式中最关键的质量保障机制。其理论依据来自：

1. **HWAT HIL-4委托级**：人类参与度5-20%，但覆盖高风险决策节点
2. **Human-in-the-Loop最佳实践**（2026年行业共识）：Approval queues、Confidence thresholds、Draft mode UX、Decision trails、Escalation triggers五种标准模式
3. **增强工作流门禁**：在工作流中设置不可绕过的质量门禁
4. **韧性工程四大能力**：响应/监控/学习/预期能力与卡点的映射（详见§3.2.6）

#### 2.3.2 三阶段硬性卡点设计

**卡点1：需求确认门禁（Requirement Confirmation Gate）**

| 维度 | 说明 |
|------|------|
| **触发时机** | 需求分析Agent完成需求拆解和优先级排序后 |
| **卡点交付物** | **AES工件**：requirement_packet.aes.yaml（机器权威工件）；**派生件**：需求分析报告摘要（Markdown，供人类评审） |
| **用户动作** | 确认/修改/补充需求 |
| **卡点强度** | 🔴 强制——不可跳过，不可被Agent自动通过 |
| **韧性能力映射** | 监控能力：需求分析进度和准确率 |
| **超时处理** | 超时24小时未响应→暂停工作流，发送提醒；超时72小时→归档 |

**卡点2：方案评审门禁（Solution Review Gate）**

| 维度 | 说明 |
|------|------|
| **触发时机** | 方案设计Agent完成技术方案和工作流编排后 |
| **卡点交付物** | **AES工件**：solution_packet.aes.yaml（机器权威工件）；**派生件**：方案设计文档摘要（Markdown，供人类评审） |
| **用户动作** | 批准/要求修改/要求简化方案 |
| **卡点强度** | 🔴 强制——不可跳过，不可被Agent自动通过 |
| **韧性能力映射** | 预期能力：方案风险预测和提前预警 |
| **超时处理** | 同卡点1 |

**卡点3：成果验收门禁（Deliverable Acceptance Gate）**

| 维度 | 说明 |
|------|------|
| **触发时机** | 执行Agent完成全部子任务，质量验证Agent确认质量达标后 |
| **卡点交付物** | 最终交付物 + 执行报告（AES工件）；**派生件**：执行摘要（Markdown，供人类评审） |
| **用户动作** | 验收通过/要求修改/退回重做 |
| **卡点强度** | 🔴 强制——不可跳过，不可被Agent自动通过 |
| **韧性能力映射** | 响应能力：交付物不达标时的自动修复；学习能力：历史数据分析 |
| **超时处理** | 同卡点1 |

#### 2.3.3 卡点与HIL分级的动态关系

```
HIL L5 (完全自主)
    │  仅当连续10次任务全部在卡点3一次通过时
    │  PDA_Score ≥ 85 且审计通过率 > 95%
    ▼
HIL L4 (委托级) ← "一句话需求"的默认层级
    │  卡点1-3全部激活，用户必须在每个卡点评审
    ▼
HIL L3 (协作级)
    │  卡点1-3激活 + 执行过程中关键步骤也需确认
    ▼
HIL L2 (监督级)
    │  全程监控 + 卡点1-3 + 额外中间检查点
    ▼
HIL L1 (辅助级)
    │  Agent仅提供建议，所有决策由人类做出
```

> **核心原则**：HIL级别决定卡点密度，但三个硬性卡点在任何HIL级别下（L1-L4）都不可关闭。只有在L5级且满足严格条件（PDA_Score、连续通过率、审计通过率）时，卡点可转为自动审核模式（Agent代审，但仍保留人类抽查权）。

### 2.4 交付物类型体系

#### 2.4.1 交付物分类

| 交付物类型 | 典型场景 | 核心Agent组合 | 质量标准 |
|-----------|---------|-------------|---------|
| **文档交付** | 研究报告、分析文档、方案设计 | doc-chunker → doc-summarizer → literature-reviewer | 格式规范、引用完整、逻辑清晰 |
| **可部署软件** | Web应用、API服务、微服务 | code-generator → skill-orchestrator → auto-logger | 通过自动化测试、代码质量达标、可部署 |
| **桌面应用** | 跨平台桌面工具 | code-generator → skill-orchestrator | 可打包、可安装、跨平台兼容 |
| **数据工程** | ETL流程、数据处理管道 | code-generator → script-optimizer | 数据一致性、性能达标、可监控 |
| **配置/基础设施** | CI/CD流水线、容器编排 | code-generator → workflow-gatekeeper | 可执行、幂等、可回滚 |
| **混合交付** | 文档+代码+配置的组合 | 多Agent协作（Task/Subagent架构） | 各子交付物满足各自质量标准 |

#### 2.4.2 三类交付物层级（v3新增）

基于 3.9.3 审查结论，将工作流中的交付物区分为三个层级：

| 层级 | 名称 | 格式 | 用途 | 优先级 |
|------|------|------|------|--------|
| **第一层** | 机器权威工件 | `.aes.yaml` | 编排引擎读取、断点恢复、跨Agent交接 | 最高 |
| **第二层** | 人类审阅件 | `.md` | 人类评审、团队沟通、知识沉淀 | 中 |
| **第三层** | 审计回执 | `.audit.json` | 合规审计、责任归因、事后追溯 | 附属 |

> **原则**：AES是工作流中的权威任务工件。Markdown从AES自动派生，仅供人类阅读和评审。任何信息冲突以AES为准。

**相变决策函数**（准形式化表达）：

```
Architecture = f(Complexity, Standardization, Knowledge, Risk)
```

**相变阈值与迟滞区间**（与 theory_whitepaper_v2.2 §5.1-5.5 保持一致，标注为"经验假设-待校准"）：

```
Effective_Complexity = w₁×T + w₂×(1-S) + w₃×(1-K)

- T: 任务复杂度（七维度加权评分，1-10分）
- S: 标准化程度（0-1）
- K: 知识结构化程度（0-1）
- 默认权重：w₁=0.5, w₂=0.3, w₃=0.2
```

| 审计通过率 | 30天 | 100次 | >95% |

**"一句话需求"的HIL定位**：

系统默认以HIL L4（委托级）运行——用户在三个硬性卡点参与评审（约占人类决策的5-20%），其余过程由Agent自主完成。系统通过PDA测量机制，根据执行历史自动调整HIL级别。

#### 3.2.3 分层标准化理论

**三层架构定义**：

```
┌─────────────────────────────────────────────────────────┐
│  核心层（Core Layer）- 🔴 绝对不能变                     │
│  • 安全边界定义、治理框架、通信接口标准、身份标识系统      │
│  • HIL-5分级标准、硬性卡点定义、审计日志不可篡改性         │
├─────────────────────────────────────────────────────────┤
│  结构层（Structure Layer）- 🟡 建议标准化，但可配置       │
│  • 目录结构、配置文件格式、文档结构、建议组织方式         │
│  • 架构相变阈值（可校准）、迟滞区间、技能库组织           │
├─────────────────────────────────────────────────────────┤
│  实现层（Implementation Layer）- 🟢 完全可演化           │
│  • 内部逻辑、工作流细节、记忆策略、资源管理               │
│  • Task/Subagent执行模式、上下文管理策略                 │
└─────────────────────────────────────────────────────────┘
```

**分层审批策略**：

| 演化层级 | 审批流程 | 审批人 | 审批时间 | 回滚要求 |
|----------|----------|--------|----------|----------|
| **实现层演化** 🟢 | 自动化/快速审批 | 治理Agent | <1小时 | 可选回滚 |
| **结构层演化** 🟡 | 技术评审 | 技术评审委员会 | <1天 | 必须可回滚 |

#### 3.2.4 博弈论边界协商与动态边界涌现

基于文献3.2.4和3.2.5的综合研究：

**核心命题**：**Agent边界应是博弈结果而非博弈前提**。边界应在Agent协作过程中通过博弈动态涌现。

**边界熵监控**：

```
H_B = -∑_{i=1}^{n} p_i log₂(p_i)
其中 p_i = skill_count_in_boundary_i / total_skills
```

**动态边界涌现的三类博弈组合**：
1. **联盟博弈**（Shapley值）：用于任务分配
2. **拍卖博弈**：用于资源竞争
3. **演化博弈**：用于长期协作模式选择

**协商协议四阶段**：需求声明 → 博弈分析 → 资源分配 → 协议执行

#### 3.2.5 上下文隔离与选择性共享

基于文献1.9的研究：

| 范式 | 性能基准 | 适用场景 | "一句话需求"中的角色 |
|------|---------|---------|-------------------|
| **强制隔离** | 100%（基准） | 安全敏感场景 | 核心层🔴数据 |
| **完全共享** | 85%（-15%） | 简单协作 | 不推荐 |
| **选择性共享** | 135%（+35%） | 复杂协作（2026主流） | **默认策略** |

**选择性共享在本框架中的应用**：
- **全局Harness共享区**：需求分析结果、方案设计文档、交付物规范——所有Agent可读
- **边界内私有区**：各Agent的内部决策过程、中间结果——仅自身可访问
- **协商临时共享区**：A2A协作时临时创建的共享上下文——协作结束后自动回收

**关键发现**：当Agent数量>5时，完全共享模式性能下降50%。选择性共享是多Agent高效协作的关键保障。

#### 3.2.6 韧性工程整合

基于文献2.7.9的研究，将韧性工程四大核心能力映射到硬性卡点机制：

**韧性工程四大能力定义**：

| 能力 | 定义 | 时间维度 | 关键活动 |
|------|------|----------|----------|
| **响应 (Responding)** | 对已发生扰动的应对能力 | 事后/事中 | 应急处理、资源调配、功能恢复 |
| **监控 (Monitoring)** | 对系统状态与环境的感知能力 | 持续 | 状态监测、异常检测、趋势分析 |
| **学习 (Learning)** | 从经验中改进的能力 | 事后 | 经验总结、知识积累、流程优化 |
| **预期 (Anticipating)** | 对潜在扰动的预见能力 | 事前 | 风险评估、场景规划、预案准备 |

**四大能力与硬性卡点的映射**：

| 韧性能力 | 卡点映射 | 具体实现 |
|---------|---------|---------|
| **响应能力** | 卡点异常处理 | • 卡点1异常：需求澄清失败→自动访谈增强<br>• 卡点2异常：方案设计失败→回退机制<br>• 卡点3异常：交付物不达标→自动修复 |
| **监控能力** | 卡点状态监控 | • 需求分析进度和准确率<br>• 方案设计进度和质量指标<br>• 执行进度和交付物质量<br>• 实时仪表盘、告警系统 |
| **学习能力** | 卡点历史分析 | • 卡点通过率趋势分析<br>• 常见问题模式识别<br>• 阈值校准算法<br>• 案例库更新 |
| **预期能力** | 卡点风险预测 | • 预测可能的卡点失败<br>• 提前预警通知用户<br>• 预防性策略调整 |

**Safety-I与Safety-II范式整合**：

| 范式 | 核心思想 | 在"一句话需求"中的应用 |
|------|---------|-------------------|
| **Safety-I** | 安全=尽可能少的事情出错 | 异常检测、故障预防、卡点拦截 |
| **Safety-II** | 安全=尽可能多的事情成功 | 成功模式学习、最佳实践推广、韧性增强 |

> **建议**：系统应同时采用 Safety-I 和 Safety-II 双重视角，既预防失败，也确保成功。

**韧性能力成熟度模型**：

| 等级 | 名称 | 特征 |
|------|------|------|
| L1 | 初始级 | 被动响应、无系统监控 |
| L2 | 可重复级 | 有监控、响应流程文档化 |
| L3 | 已定义级 | 主动监控、定期演练、经验复盘 |

### 3.3 协议与协作体系：MCP、A2A与AES（v3重写）

#### 3.3.1 三层分工：运行时协议与任务级载体

> **核心定位**：本框架的协议与协作体系由三个互补层次组成，各自职责清晰，不存在功能重叠：

| 层次 | 名称 | 职责范围 | 对应标准/规范 | 作用对象 |
|------|------|---------|-------------|---------|
| **通信层** | A2A (Agent-to-Agent Protocol) | Agent间运行时消息协作与任务委派 | Google A2A Specification v0.3.0 | Agent ↔ Agent |

#### 3.3.4 AES：任务级协作载体（v3新增）

##### 3.3.4.1 AES的定义与定位

**AES（Agent-Executable Specification）** 是一种面向长任务、多阶段、可中断环境的机器可读任务规范。其角色接近"**任务简报 + 交接文档 + 进度状态 + 下一位Agent启动指令**"的统一载体。

**AES不是协议**。AES不负责Agent间的消息传递（那是A2A的职责）或工具连接（那是MCP的职责）。AES负责的是：

1. **任务状态承载**：记录任务当前状态、已完成步骤、待办事项
2. **上下文连续性**：保存恢复执行所需的最小上下文
3. **断点恢复**：使任意Agent可从精确断点继续工作
4. **跨Agent交接**：作为Agent间任务转交的标准化载体
5. **输入输出契约**：定义节点的input_contract和output_contract
6. **治理信息**：记录所有权、审批回执、工件哈希等治理字段

##### 3.3.4.2 AES三层结构

AES采用三层结构（与 AES_Planning_Document_Writing_Guide.aes.yaml 和 AES README 保持一致）：

| 层 | 名称 | 内容 | 在工作流中的角色 |
|----|------|------|----------------|
| **Layer 1** | AES Core | 任务起源、约束条件、成功标准、风险 | 接手Agent快速理解"为什么做这个任务" |
| **Layer 2** | Implementation Instructions | 给接手Agent的执行指令、环境检查、恢复步骤 | 接手Agent快速理解"怎么做"和"从哪里开始" |
| **Layer 3** | Plan & Artifacts | 具体任务分解、交付物定义、实施报告 | 接手Agent快速理解"做了什么"和"还剩什么" |

##### 3.3.4.3 AES与DACES/HWAT的理论映射

AES作为任务级协作载体，与DACES理论的工作流原子模型（theory_whitepaper_v2.2 §6.2）形成自然映射：

| AES字段 | DACES/HWAT映射 | 说明 |
|---------|---------------|------|
| `status.current_phase` | DACES S1/S2 状态 | 任务当前处于稳定执行还是适应演化 |
| `evolution.hil_level` | HWAT HIL-5 | 当前任务的HIL级别 |
| `evolution.architecture_mode` | DACES Script/Skill/Agent/Multi-Agent | 当前架构模式 |
| `intent.constraints` | 核心层🔴约束 | 不可违反的硬性约束 |
| `governance.risk_level` | Control Plane 风险策略 | 风险等级与审批要求 |
| `next_agent_context` | HWAT 工作流原子模型 L1/L2 | 跨Agent交接的最小上下文 |
| `artifacts.checkpoint_receipts` | 韧性工程监控能力 | 卡点审批记录与审计 |
| `artifacts.decisions_made` | AEAS 决策核 | 关键决策记录与追溯 |

##### 3.3.4.4 AES在企业工作流中的关键特性

### 4.3 HIL动态调整与PDA测量体系

#### 4.3.1 HIL动态调整算法

```python
class HILDynamicAdjuster:
    def adjust_hil_level(self, task: Task, history: ExecutionHistory, pda_score: float) -> HILLevel:
        """
        基于PDA评分和执行历史动态调整HIL级别
        
        与 theory_whitepaper_v2.2 §8A.1.3 保持一致
        """
        score = self._calculate_hil_score(task, history, pda_score)
        
        if score >= 90:
            return HILLevel.L5
        elif score >= 75:
            return HILLevel.L4
        elif score >= 60:
            return HILLevel.L3
        elif score >= 40:
            return HILLevel.L2
        else:
            return HILLevel.L1
    
    def _calculate_hil_score(self, task: Task, history: ExecutionHistory, pda_score: float) -> float:
        weights = {
            'completion_rate': 0.20,
            'intervention_rate': 0.15,
            'exception_recovery': 0.15,
            'audit_pass_rate': 0.15,
            'checkpoint_pass_rate': 0.15,
            'pda_score': 0.10,
            'risk_level': 0.10,
        }
        
        risk_level_map = {'L1': 1, 'L2': 2, 'L3': 3, 'L4': 4, 'L5': 5}
        risk_level_score = risk_level_map.get(task.risk_level, 3)
        
        score = (
            weights['completion_rate'] * history.completion_rate * 100 +
            weights['intervention_rate'] * (1 - history.intervention_rate) * 100 +
            weights['exception_recovery'] * history.exception_recovery * 100 +
            weights['audit_pass_rate'] * history.audit_pass_rate * 100 +
            weights['checkpoint_pass_rate'] * history.checkpoint_first_pass_rate * 100 +
            weights['pda_score'] * pda_score +
            weights['risk_level'] * (5 - risk_level_score) * 20
        )
        
        return min(max(score, 0), 100)
```

#### 4.3.2 PDA测量体系（与theory_whitepaper_v2.2 §8A.1.2一致）

**PDA指标簇**（4大类16小类）：

**A. 任务自治指标（Task Autonomy Metrics）**

| 指标 | 定义 | 计算方式 | 目标值（经验假设） | 适用HIL等级 |
|------|------|---------|------------------|-------------|
| 任务独立完成率 | 无需人类干预完成的任务比例 | 独立完成任务数 / 总任务数 × 100% | L3: >70%, L4: >90% | L3-L5 |
| 人类干预率 | 人类介入任务执行的比例 | 干预次数 / 任务数 | L3: <30%, L4: <10% | L2-L5 |
| 任务重规划频率 | 任务执行中重新规划的比例 | 重规划次数 / 任务数 | <20% | L3-L5 |
| 工具调用自主率 | Agent自主决定工具调用的比例 | 自主调用数 / 总调用数 × 100% | L3: >60%, L4: >85% | L3-L5 |

**B. 轨迹质量指标（Trajectory Quality Metrics）**

| 指标 | 定义 | 计算方式 | 目标值 |
|------|------|---------|-------|
| 任务轨迹完整率 | 完整记录执行步骤的比例 | 完整轨迹数 / 总任务数 × 100% | >95% |
| 无效步骤占比 | 不贡献于目标达成的步骤比例 | 无效步骤数 / 总步骤数 × 100% | <10% |
| 工具选择准确率 | 选择正确工具的比例 | 正确选择数 / 总选择数 × 100% | >90% |
| 失败恢复成功率 | 失败后成功恢复的比例 | 成功恢复数 / 失败总数 × 100% | >80% |

**C. 风险控制指标（Risk Control Metrics）**

| 指标 | 定义 | 计算方式 | 目标值 | 优先级 |
|------|------|---------|-------|-------|
| 高风险操作拦截率 | 被控制平面拦截的高风险操作比例 | 拦截数 / 高风险操作总数 × 100% | 100% | P0 |
| 审计缺失率 | 缺少审计日志的操作比例 | 缺失日志数 / 总操作数 × 100% | 0% | P0 |
| 越权调用率 | 超出权限的调用比例 | 越权次数 / 总调用数 × 100% | 0% | P0 |
| 安全策略违反率 | 违反安全策略的比例 | 违反次数 / 总操作数 × 100% | <0.1% | P0 |

**D. 组织协同指标（Organizational Coordination Metrics）**

| 指标 | 定义 | 计算方式 | 目标值 |
|------|------|---------|-------|
| 人类注意力消耗 | 人类监督花费的时间占比 | 监督时间 / 任务总时长 × 100% | L3: <40%, L4: <15% |
| 升级到人工的响应时延 | 从请求人工介入到实际响应的平均时间 | 平均响应时间 | <5分钟（高风险） |
| 责任归因完成率 | 可追溯到具体责任人的决策比例 | 可归因决策数 / 总决策数 × 100% | 100% |
| 人工兜底成功率 | 人工接管后成功解决问题的比例 | 成功数 / 接管总数 × 100% | >95% |

**PDA综合评分公式**：

```
PDA_Score = w₁·Task_Autonomy + w₂·Trajectory_Quality + w₃·Risk_Control + w₄·Human_Oversight

默认权重：
- w₁ (Task_Autonomy): 0.35
- w₂ (Trajectory_Quality): 0.25
- w₃ (Risk_Control): 0.25
- w₄ (Human_Oversight): 0.15

说明：权重为策略配置项，不同场景可调整权重
```

### 4.4 架构相变决策树

基于文献3.2.1的三层相变统一理论：

```
当前架构：Script
├─ IF 标准化<70% OR 异常分支>5 → 切换到 Skill
└─ ELSE → 保持 Script

当前架构：Skill

当前架构：Multi-Agent
├─ IF 复杂度<5 AND 领域<2 → 回退到 Agent
```


#### 4.7.3 断点恢复机制

基于AES的断点恢复流程：

```
工作流执行中 → 任意节点中断
    ↓
1. 编排引擎检测到中断（超时/错误/人工终止）
    ↓
2. 将当前节点状态快照写入AES工件：
   - status.completion_percentage = 当前完成度
   - status.completed_items = [已完成的步骤]
   - status.pending_items = [待执行的步骤]
   - next_agent_context.critical_assumptions = [关键假设]
   - next_agent_context.hard_constraints = [不可违反的约束]
   - next_agent_context.relevant_specs = [相关规格引用]
    ↓
3. AES工件持久化存储（WORM策略）
    ↓
4. 恢复时：
   a. 新Agent（或同一Agent新会话）读取AES工件
   b. 检查 next_agent_context 获取恢复上下文
   c. 从 last_completed_step 的下一步开始执行
   d. 无需重新理解整个项目历史
```

**恢复策略配置**：

```yaml
resume_policy:
  auto_resume:
    enabled: true
    conditions:
      - "interruption_type == 'session_timeout'"
      - "interruption_type == 'model_switch'"
      - "time_since_interruption < 24h"
  
  manual_resume:
    required_for:
      - "interruption_type == 'human_terminate'"
      - "time_since_interruption >= 24h"
    approval_required: true
    
  state_validation:
    on_resume:
      - "verify_input_contract_still_valid"
      - "check_dependency_artifacts_exist"
      - "confirm_no_schema_migration_needed"
```

#### 4.7.4 执行引擎核心组件

1. **工作流解析器**：解析YAML工作流定义，验证DAG无环，解析节点五元组
2. **Harness管理器**：管理全局和Agent级Harness的创建、配置和销毁
3. **上下文管理器**：维护选择性共享上下文，管理临时共享区的生命周期
4. **卡点管理器**：管理硬性卡点的触发、状态转换、超时处理、韧性能力映射、审批回执生成
5. **Task/Subagent管理器**：管理Subagent的创建、执行、结果聚合
6. **A2A协调器**：管理Agent间的发现、通信、任务委派和结果聚合
7. **MCP代理层**：聚合多个MCP Server，提供统一的工具调用接口
8. **边界熵监控器**：实时监控边界熵，触发动态边界协商
9. **PDA测量器**：持续采集PDA指标，支持HIL动态调整
10. **可观测性层**：OpenTelemetry集成，分布式追踪、指标采集、审计日志
11. **状态持久化**：支持工作流状态保存与恢复（通过AES工件）
12. **AES工件管理器**：管理AES工件的创建、版本控制、交接、哈希校验（v3新增）
13. **工件注册表**：跟踪所有工件（AES/Markdown/Audit）的创建、引用和完整性（v3新增）
14. **Durable Execution 引擎**：管理持久化执行模式（exit/async/sync）、确定性重放、Checkpointer 和 Saga 补偿（v4新增）

### 4.8 Durable Execution 与一致性重放（v4新增）

#### 4.8.1 定义与定位

**Durable Execution（持久化执行）** 是一种运行时语义，确保工作流进程在关键检查点保存执行状态，使进程在中断（系统故障、模型切换、人工介入）后能够从精确断点恢复，且不重复已完成的有副作用操作。

> **核心区分**：Durable Execution 是**运行时语义**（runtime semantics），AES 是**任务工件**（task artifact）。Durable Execution 关注"进程如何可靠地执行和恢复"，AES 关注"任务状态如何被承载和交接"。两者互补但不可互相替代。

**理论基础与外部参考**：

| 来源 | 核心概念 | 本框架应用 |
|------|---------|-----------|
| LangGraph Durable Execution（LangChain, 2026） | durability modes（exit/async/sync）、deterministic replay、checkpointer、task 副作用隔离 | 工作流节点持久化模式设计 |
| Temporal（temporal.io） | durable workflows、compensation/saga patterns、pause/resume、Activities | 补偿/Saga 模式、长时运行工作流保障 |
| HWAT HIL-4 委托级 | 人工介入审批与恢复 | Human-in-the-Loop 与持久化执行的集成 |

#### 4.8.2 三级持久化模式

借鉴 LangGraph 的 durability modes，本框架定义三个持久化级别，平衡性能与数据一致性：

| 模式 | 名称 | 行为 | 性能开销 | 一致性保证 | 适用场景 |
|------|------|------|---------|-----------|---------|
| **exit** | 退出持久化 | 仅在节点正常退出、异常终止或触发人工审批时保存状态 | 最低 | 中间状态不保存，进程崩溃则丢失未保存进度 | 长时间低风险工作流 |
| **async** | 异步持久化 | 在下一步执行的同时异步保存当前状态 | 低 | 极小概率丢失最后一步状态（进程崩溃时） | 大多数生产工作流（默认推荐） |
| **sync** | 同步持久化 | 在下一步开始前同步保存当前状态 | 中 | 每个检查点都确保写入成功后再继续 | 高风险/高价值工作流 |

```yaml
durable_execution:
  default_mode: "async"
  per_node_override:
    - node_id: "node_req_analysis"
      mode: "sync"
      reason: "需求分析结果为关键工件，不允许丢失"
    - node_id: "node_solution_design"
      mode: "sync"
      reason: "方案设计为高风险决策节点"
    - node_id: "node_execution"
      mode: "async"
      reason: "执行阶段追求性能"
    - node_id: "node_quality"
      mode: "sync"
      reason: "质量验证结果为交付门禁依据"
```

#### 4.8.3 确定性重放（Deterministic Replay）

当工作流从中断点恢复时，执行引擎不会从"同一行代码"恢复，而是从一个适当的起始节点开始，重放该节点到中断点之间的所有步骤。为确保重放的一致性，以下原则必须遵守：

**确定性重放三原则**：

1. **避免重复工作**：包含多个有副作用操作（API调用、文件写入、网络请求）的节点，应将每个操作封装为独立的 **Activity**（参考Temporal），确保恢复时从持久化层获取已记录的结果而非重新执行
2. **封装非确定性操作**：随机数生成、时间戳等非确定性操作必须封装在 Activity 内，恢复时返回记录值
3. **使用幂等操作**：尽可能确保副作用操作是幂等的（同一操作执行多次结果相同），使用幂等键（idempotency key）避免重复执行

```
工作流执行：节点A → 节点B(步骤1, 步骤2, 步骤3) → 节点C
                    ↓ 中断
恢复时：
  1. 引擎识别中断点位于节点B的步骤2之后
  2. 从节点B起始位置开始重放
  3. 步骤1：Activity结果已持久化 → 直接返回记录值（不重复执行）
  4. 步骤2：Activity结果已持久化 → 直接返回记录值（不重复执行）
  5. 步骤3：未执行 → 正常执行
  6. 继续到节点C
```

**Checkpointer 机制**：

```yaml
checkpointer:
  storage: "durable_store"
  thread_identifier:
    format: "wf_{workflow_id}_node_{node_id}_v{version}"
    scope: "per_workflow_instance"
  state_snapshot:
    includes:
      - "node_state"
      - "activity_results"
      - "aes_artifact_hash"
      - "dependency_outputs"
    on_interrupt:
      - "save_full_snapshot"
    on_activity_complete:
      - "save_activity_result"
```

#### 4.8.4 补偿模式与 Saga（Compensation/Saga）

对于已执行但需要回滚的操作序列，本框架引入补偿模式（参考Temporal的Saga模式）：

**Saga 补偿流程**：

```
工作流：步骤A → 步骤B → 步骤C → 步骤D
                           ↓ 步骤D失败
触发 Saga 补偿：
  步骤C.compensate() → 步骤B.compensate() → 步骤A.compensate()
  （逆序补偿，确保数据一致性）
```

**补偿策略配置**：

```yaml
compensation:
  strategy: "saga"
  execution:
    order: "reverse"
    timeout_per_step: 300
    max_retries: 3
    retry_backoff: "exponential"
  node_compensation_map:
    - node_id: "node_execution"
      compensate_by: "execution_rollback_agent"
      compensable_artifacts:
        - "generated_code"
        - "config_files"
    - node_id: "node_solution_design"
      compensate_by: "design_revert_agent"
      compensable_artifacts:
        - "architecture_decisions"
        - "dependency_changes"
  non_compensable_nodes:
    - "node_req_analysis"
    - "node_intent"
    description: "需求确认和意图理解为不可逆节点"
    handling: "标记为部分完成，需人工决策"
```

#### 4.8.5 Durable Execution 与 AES 的关系

| 维度 | Durable Execution | AES |
|------|------------------|-----|
| **本质** | 运行时进程语义 | 任务级数据工件 |
| **关注点** | 进程如何可靠执行和恢复 | 任务状态如何承载和交接 |
| **持久化对象** | 执行状态快照、Activity结果 | 任务简报、约束条件、进度状态 |
| **恢复机制** | Checkpointer + 确定性重放 | 读取 AES 工件 → next_agent_context |
| **补偿能力** | Saga 补偿模式 | 无（AES 不执行操作） |
| **持久化模式** | exit/async/sync | 始终持久化（WORM策略） |
| **生命周期** | 节点执行期间 | 任务全生命周期 |

> **协同关系**：Durable Execution 在节点执行期间提供进程级可靠性保障；AES 在节点间提供任务级连续性保障。当节点执行中断时，Durable Execution 确保进程状态可恢复；当跨 Agent 交接时，AES 确保任务上下文可传递。两者共同构成"进程+工件"双层可靠性保障。

### 5.1 演化边界与原则

**演化边界定义**：

```
┌─────────────────────────────────────────────┐
│  演化区域：实现层（绿色）                    │
│      ↓                                       │
│  演化提案 → 风险评估 → 模拟测试 → 审批 → 实施  │

│  影响评估：PDA指标不能下降超过10%            │
└─────────────────────────────────────────────┘
```

**演化原则**：
1. **只能演化实现层**：核心层🔴绝对不能变，结构层🟡变更需额外评审
2. **必须经过审批**：所有演化都需要审批

4. **必须记录**：所有演化都要完整记录
5. **分阶段实施**：小范围测试 → 扩大范围 → 全面推广
6. **Harness约束不放松**：演化不得削弱Harness的三大护栏能力
7. **韧性能力不降低**：演化不得削弱四大韧性能力
8. **PDA指标监控**：演化后PDA指标不得显著下降
9. **AES工件一致性**：演化后AES工件Schema变更需向后兼容（v3新增）
10. **断点恢复保障**：演化不得破坏断点恢复机制（v3新增）

### 5.4 Policy-as-Code Control Plane（v4新增）

#### 5.4.1 定义与定位

**Policy-as-Code Control Plane** 将治理从内嵌于业务逻辑中的规则升级为独立的、可解耦的、可验证的声明式策略引擎。其核心目标是：**授权逻辑与业务逻辑分离，策略变更无需修改代码，策略效果可测试可审计**。

> **与现有治理的关系**：v3 §5.2 的多Agent共建治理（所有权管理、冲突解决、审批回执）是治理的**执行层**。Policy-as-Code 是治理的**控制平面层**，定义"谁在什么条件下可以对什么资源执行什么操作"的规则。

**理论基础与外部参考**：

| 来源 | 核心概念 | 本框架应用 |
|------|---------|-----------|
| Cedar Policy Language（AWS, 2022-） | principal-action-resource-context 授权模型 | 授权模型设计 |
| OPA (Open Policy Agent) | 声明式策略语言 Rego | 策略评估引擎 |
| ABAC（基于属性的访问控制） | 属性驱动的动态授权 | 策略条件设计 |

#### 5.4.2 授权模型：PARC 四元组

本框架采用 Cedar 风格的 **PARC（Principal-Action-Resource-Context）** 授权模型：

| 维度 | 定义 | 示例 |
|------|------|------|
| **P (Principal)** | 发起操作的实体 | `Agent::"requirement-analyst"`, `Human::"user_001"` |
| **A (Action)** | 尝试执行的操作 | `Action::"task.execute"`, `Action::"checkpoint.approve"`, `Action::"aes.write"` |
| **R (Resource)** | 操作的目标资源 | `AES::"req_packet_001"`, `Node::"node_execution"`, `Checkpoint::"cp1"` |
| **C (Context)** | 操作时的上下文属性 | `{"hil_level": "L3", "risk_level": "HIGH", "time": "business_hours"}` |

**策略定义示例（Cedar 风格）**：

```
permit(
  principal == Agent::"requirement-analyst",
  action in [Action::"task.execute", Action::"aes.write"],
  resource == AES::"req_packet_001"
) when {
  resource.owner == principal &&
  context.hil_level in ["L3", "L4"] &&
  context.risk_level != "CRITICAL"
};

permit(
  principal == Human::"user_001",
  action == Action::"checkpoint.approve",
  resource == Checkpoint::"cp1"
) when {
  context.audit_trail_complete == true
};

forbid(
  principal in Agent::"",
  action == Action::"checkpoint.bypass",
  resource in Checkpoint::""
) when {
  true
};
```

#### 5.4.3 策略评估点（PEP）与调用链集成

策略评估点（Policy Evaluation Point, PEP）嵌入在 A2A/MCP/AES 调用链的关键位置：

```
Agent A 通过 A2A 请求 Agent B 执行任务：
    ↓
[PEP-1: 身份验证] → Agent Card 签名验证
    ↓
[PEP-2: 授权评估] → PARC 策略引擎评估
    ↓  允许 → 继续
    ↓  拒绝 → 返回 DENIED + 审计日志
Agent B 通过 MCP 调用工具：
    ↓
[PEP-3: 工具授权] → 检查 Agent B 是否有权限调用该工具
    ↓
Agent B 更新 AES 工件：
    ↓
[PEP-4: 工件授权] → 检查 Agent B 是否是当前工件所有者
    ↓
Agent B 触发卡点审批：
    ↓
[PEP-5: 卡点策略] → 检查审批人权限和审计完整性
```

#### 5.4.4 策略生命周期管理

| 阶段 | 活动 | 审批要求 |
|------|------|---------|
| **草拟** | 策略编写、单元测试 | 无 |
| **审查** | 策略影响分析、dry-run 评估 | 技术评审 |
| **发布** | 部署到策略引擎 | 结构层🟡审批 |
| **审计** | 运行时效果监控、定期审查 | 自动化 |
| **退役** | 标记废弃、清理引用 | 结构层🟡审批 |

> **治理增强原则**：Policy-as-Code 不取代 v3 §5.2 的多Agent共建治理，而是在其之上提供更精细化的授权控制。所有权管理（§5.2.1）定义"谁拥有什么"，Policy-as-Code 定义"谁能对什么做什么"。

### 5.5 Eval-Driven Control Plane（v4新增）

#### 5.5.1 定义与定位

**Eval-Driven Control Plane** 将评估（Evaluation）从附属指标升级为控制平面的一等公民。评估结果不仅被记录和展示，还直接驱动门禁决策、配置晋升和自动回滚。

> **与 PDA 的关系**：PDA 测量体系（§4.3.2）提供**部署后自治能力的持续测量**。Eval-Driven Control Plane 在 PDA 基础上增加**评估驱动的控制决策能力**，使评估结果能够触发自动化动作。

**理论基础**：

| 来源 | 核心概念 | 本框架应用 |
|------|---------|-----------|
| OTel GenAI Events — `gen_ai.evaluation.result` | 评估结果事件规范 | 评估结果记录格式 |
| ML 模型评估最佳实践 | 离线/在线/影子评估三模式 | 评估模式设计 |
| Canary Deployment | 灰度发布与自动回滚 | 晋升与回滚门禁 |

#### 5.5.2 三种评估模式

| 模式 | 名称 | 执行时机 | 数据来源 | 决策影响 |
|------|------|---------|---------|---------|
| **Offline Eval** | 离线评估 | 代码/配置变更前 | 历史数据集、测试套件 | 阻止不达标变更进入生产 |
| **Online Eval** | 在线评估 | 生产运行中 | 实时执行数据 | 触发告警、自动降级 |
| **Shadow Eval** | 影子评估 | 与生产并行 | 生产流量副本 | 对比新旧版本效果，驱动晋升决策 |

**评估模式配置**：

```yaml
eval_control_plane:
  modes:
    offline:
      enabled: true
      trigger: "pre_deployment"
      gate_threshold: 0.8
      metrics:
        - name: "task_completion_accuracy"
          weight: 0.4
        - name: "intent_resolution_quality"
          weight: 0.3
        - name: "code_quality_score"
          weight: 0.3
          
    online:
      enabled: true
      trigger: "continuous"
      evaluation_window: "1h"
      alert_threshold: 0.7
      auto_degrade: true
      degrade_to_hil: "L3"
      
    shadow:
      enabled: true
      trigger: "on_candidate_promotion"
      traffic_percentage: 10
      comparison_window: "24h"
      promotion_criteria:
        min_sample_size: 100
        min_improvement: 0.05
        max_regression: 0.02
```

#### 5.5.3 评估驱动门禁

**评估结果事件格式（对齐 OTel gen_ai.evaluation.result）**：

```json
{
  "event_name": "gen_ai.evaluation.result",
  "timestamp": "2026-03-28T10:00:00Z",
  "attributes": {
    "gen_ai.evaluation.name": "TaskCompletionAccuracy",
    "gen_ai.evaluation.score.value": 0.92,
    "gen_ai.evaluation.score.label": "pass",
    "gen_ai.evaluation.explanation": "Task completed within quality thresholds",
    "gen_ai.conversation.id": "conv_001",
    "eval.mode": "online",
    "eval.gate": "cp3_acceptance"
  }
}
```

**门禁决策矩阵**：

| 评估结果 | 门禁动作 | 示例 |
|---------|---------|------|
| score ≥ promotion_threshold | 通过门禁，允许晋升 | Online eval 0.92 → 通过交付门禁 |
| score ≥ alert_threshold | 通过门禁，发送观察通知 | Online eval 0.75 → 通知运维 |
| score < alert_threshold | 阻止通过，触发调查 | Online eval 0.65 → 阻止交付，自动降级HIL |
| shadow_score > current_score | 候选版本晋升 | Shadow eval 候选0.90 > 当前0.85 → 晋升 |
| shadow_score ≤ current_score | 候选版本保持 | Shadow eval 候选0.83 ≤ 当前0.85 → 不晋升 |

**Promotion/Rollback 流程**：

```
候选版本进入影子评估
    ↓
影子评估运行 ≥ 24h，样本量 ≥ 100
    ↓
┌─ shadow_score > current_score + min_improvement
│   → 自动晋升候选版本为生产版本
│   → 更新 PDA 基线
│   → 记录审计日志
│
└─ shadow_score ≤ current_score - max_regression
    → 阻止晋升
    → 触发告警
    → 候选版本回退到开发环境
```

> **与硬性卡点的关系**：硬性卡点（§2.3）是面向人类的评审门禁。Eval-Driven 门禁是面向系统的自动化质量门禁。两者互补：硬性卡点确保人类意图对齐，Eval-Driven 门禁确保系统质量达标。在 HIL L5（完全自主）场景下，Eval-Driven 门禁替代部分硬性卡点的人类评审职能。

#### 7.4.3 Delta Sync 机制

对于大型任务中 AES 工件频繁更新的场景，引入增量同步（Delta Sync）机制：

```yaml
delta_sync:
  strategy: "patch_based"
  patch_format: "json_patch_rfc6902"
  
  trigger_conditions:
    - "artifact_size > 100KB"
    - "update_frequency > 10_per_hour"
    
  sync_flow:
    1. "生成 base_snapshot（完整快照）"
    2. "后续更新生成 delta_patch（差异补丁）"
    3. "接收方应用 delta_patch 到本地 base_snapshot"
    4. "定期重新生成 base_snapshot（防止补丁链过长）"
    
  consistency:
    base_snapshot_interval: "every_10_patches"
    hash_verification: "sha256_per_patch"
```

随着理论体系的不断完善和工程实践的持续积累，基于动态适应与协同演化理论的多Agent系统工程应用框架有望成为企业智能化转型的核心方法论。

通过**Harness Engineering**提供执行约束，通过**分层标准化**实现可控的灵活性，通过**受控演化**实现安全的自主性，通过**硬性卡点**保障交付质量，通过**韧性工程**增强系统可靠性，通过**Task/Subagent架构**优化执行效率，通过**PDA测量**支持持续优化，通过**AES任务级协作载体**保障任务连续性和断点恢复，通过**Durable Execution**提供运行时可靠性保障，通过**Policy-as-Code**实现声明式治理控制，通过**Eval-Driven Control Plane**形成评估驱动闭环，通过**OTel GenAI 语义对齐**融入行业观测生态——最终实现人机协同的智能化工作新范式。

> **"一句话需求"不仅是交互方式的简化，更是人类与AI协作关系的根本重构：人类负责方向和评判，Agent负责路径和执行。当这个闭环通过DACES双态演化、Harness执行护栏、三阶段硬性卡点、韧性四大能力、PDA测量体系、AES任务级协作载体、Durable Execution运行时语义、Policy-as-Code声明式治理、Eval-Driven评估控制闭环和OTel GenAI观测对齐形成稳定的动态平衡时，企业工作流的智能化转型就完成了从"可能"到"可工程化"的关键跃迁。**
